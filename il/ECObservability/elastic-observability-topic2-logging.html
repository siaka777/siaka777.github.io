03
Filebeat
Configure Filebeat to Collect & Ship Logs
FilebeatConfig
Filed ✓ ▾

Filebeat is a lightweight log shipper. It reads log files from disk (or stdin, syslog, etc.), optionally processes them with modules or ingest pipelines, and ships them to Elasticsearch or Logstash. It uses inputs to define what to read and modules for pre-built parsing of common log formats.

🎯 Exam Tasks Configuring Filebeat inputs, enabling/disabling modules, setting up output to Elasticsearch, and running setup commands.
Interactive Filebeat Config Builder
Filebeat.yml Generator INTERACTIVE
← Configure and click Generate
Key Filebeat Commands
# Enable a module filebeat modules enable nginx # List all modules (enabled/disabled) filebeat modules list # Test configuration syntax filebeat test config -e # Test Elasticsearch connection filebeat test output # Setup (creates index templates, ILM policies, dashboards) filebeat setup -e # Run once (useful for testing, then exit) filebeat run -e --once # Start Filebeat service systemctl start filebeat
⚠️ Watch Out The filestream input is the modern replacement for the legacy log input. They are NOT interchangeable — the registry format differs. For new deployments, always use filestream. For existing log input configs, do not migrate without care.

Grok & Parsing
Parse Unstructured Logs with Grok Patterns
ParsingFilebeat
Filed ✓ ▾

When no module covers your log format, you use a grok processor in an ingest pipeline to parse raw log strings into structured fields. Grok uses named regex patterns to extract fields from text.

Interactive Grok Pattern Tester
Grok Pattern Tester
Click "Test Pattern" to see extracted fields →
Common Grok Patterns
# Common built-in Grok patterns (use %{PATTERN_NAME:field_name}) %{IP:client_ip} # IPv4 or IPv6 address %{IPORHOST:host} # IP or hostname %{NUMBER:status:int} # Integer number (:int = cast to integer) %{WORD:method} # Single word (no spaces) %{URIPATH:path} # URI path (/api/v1/users) %{URIPATHPARAM:req} # URI path + query params %{HTTPDATE:timestamp} # Apache/nginx date format %{GREEDYDATA:message} # Match everything (use last) %{DATA:field} # Match anything (non-greedy) %{LOGLEVEL:level} # ERROR, WARN, INFO, DEBUG etc. # Ingest pipeline using grok processor { "grok": { "field": "message", "patterns": [ "%{IP:client_ip} .* \"%{WORD:method} %{URIPATHPARAM:request}\"" ], "ignore_failure": true } }
💡 Tip Use the Kibana Dev Tools → Grok Debugger (Dev Tools → Grok Debugger in the top nav) to test patterns interactively before adding them to an ingest pipeline. Much faster than trial-and-error.
06
Stack Monitoring
Configure Filebeat for Stack Monitoring
FilebeatMonitoring
Filed ✓ ▾

Just like Metricbeat collects metrics for Stack Monitoring, Filebeat collects logs from the Elastic Stack components (Elasticsearch, Kibana, Logstash) and ships them to the monitoring cluster. This is the recommended approach for centralized stack log collection.

🎯 Exam Objective "Configure Filebeat to gather Stack Monitoring data." Enable the elasticsearch and kibana modules on Filebeat with xpack.enabled: true and point them at your stack components.
# Step 1: Enable stack monitoring modules on Filebeat filebeat modules enable elasticsearch filebeat modules enable kibana # modules.d/elasticsearch.yml — collect ES logs - module: elasticsearch server: enabled: true var.paths: - "/var/log/elasticsearch/*.log" - "/var/log/elasticsearch/*.json" gc: enabled: true var.paths: - "/var/log/elasticsearch/*_gc.log*" audit: enabled: false # enable if audit logging is on xpack.enabled: true # ← REQUIRED for Stack Monitoring # modules.d/kibana.yml - module: kibana log: enabled: true var.paths: ["/var/log/kibana/kibana.log"] xpack.enabled: true # ← REQUIRED for Stack Monitoring # Output to the MONITORING cluster (separate from production) output.elasticsearch: hosts: ["https://monitoring-es:9200"] username: "remote_monitoring_user" password: "YOUR_PASSWORD"
💡 Key Detail xpack.enabled: true is the critical setting that makes a Filebeat module write to the Stack Monitoring indices (.monitoring-*) instead of the regular Filebeat indices. Without it, data won't appear in Stack Monitoring.
Practice Quiz
■ Exam Question

Multiline & Advanced
Multiline Logs, Custom Inputs & Processors
FilebeatAdvanced
Filed ✓ ▾

Some log formats span multiple lines (Java stack traces, Python tracebacks). Filebeat's multiline configuration stitches these into a single event. Additionally, processors let you enrich or filter events before shipping.

# Multiline config — Java stack traces filebeat.inputs: - type: filestream id: my-java-app-logs paths: - /var/log/myapp/*.log multiline.type: pattern multiline.pattern: '^[[:space:]]+(at|\.{3})\b|^Caused by:' multiline.negate: false # lines matching pattern = continuation multiline.match: after # append to previous line # Python tracebacks — starts with "Traceback" multiline.type: pattern multiline.pattern: '^Traceback' multiline.negate: true # new event = line NOT matching pattern multiline.match: after # Filebeat processors processors: - add_host_metadata: when.not.contains.tags: forwarded - add_cloud_metadata: ~ - add_docker_metadata: ~ - add_kubernetes_metadata: ~ - drop_fields: fields: ["agent.ephemeral_id", "ecs.version"] ignore_missing: true
💡 Multiline Logic Think of it as: negate: false = "this pattern IS a continuation line" (attach to previous). negate: true = "this pattern is the START of a new event" (lines that DON'T match get appended to it).
Practice Quiz
■ Exam Question
Java stack traces start with a timestamp line, then have continuation lines beginning with whitespace or "at". Which multiline config correctly groups them into one event?
Apattern: '^[[:space:]]', negate: false, match: after
Bpattern: '^[[:space:]]', negate: true, match: before
Cpattern: '^\d{4}-', negate: false, match: after
Dpattern: '^\d{4}-', negate: true, match: before
