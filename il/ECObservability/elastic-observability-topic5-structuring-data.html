Ingest Node
Define and Use Ingest Pipelines
PipelineElasticsearchCore Exam
✓
▾

An ingest pipeline is a sequence of processors applied to documents before they are indexed into Elasticsearch. Pipelines run on ingest nodes (all nodes by default). Define them in Kibana → Stack Management → Ingest Pipelines, or via the API.

🎯
Exam objective: Create a pipeline, add processors, test with a sample document, and apply it to an index or Beats output.
Create & Simulate an Ingest Pipeline
# Create a pipeline PUT _ingest/pipeline/parse-nginx-logs { "description": "Parse Nginx access logs", "processors": [ { "grok": { "field": "message", "patterns": ["%{COMBINEDAPACHELOG}"] } }, { "date": { "field": "timestamp", "formats": ["dd/MMM/yyyy:HH:mm:ss Z"] } }, { "geoip": { "field": "clientip" } }, { "remove": { "field": "message" } } ] } # Test before applying POST _ingest/pipeline/parse-nginx-logs/_simulate { "docs": [{ "_source": { "message": "198.51.100.1 - - [27/Feb/2026:10:30:00 +0000] \"GET /api HTTP/1.1\" 200 142" } }] }
Applying Pipelines
MethodHow to apply
Index API?pipeline=<name> appended to indexing request
Index settingsindex.default_pipeline or index.final_pipeline
Beats outputoutput.elasticsearch.pipeline: "<name>"
Elastic AgentSet pipeline in integration or component template settings
💡
index.default_pipeline runs on every doc. index.final_pipeline always runs last, even after per-request pipelines — ideal for global enrichment.

Grok is a pattern-matching language built on named regex captures. Patterns use the syntax %{PATTERN_NAME:field_name} or %{PATTERN:field:type} for type coercion. 100+ built-in patterns available.

💡
Dissect vs Grok: Use dissect when log format is consistent and delimiter-based (much faster). Use grok for variable-format logs needing regex flexibility.
04
Logstash
Configure Logstash Pipelines for Data Processing
LogstashElasticsearch
✓
▾

Logstash is a server-side processing pipeline with inputs, filters, and outputs. More powerful than ingest pipelines for complex routing and fan-out, but adds operational overhead. Its pipeline DSL uses input, filter, and output blocks.

Logstash Pipeline Configuration
# /etc/logstash/conf.d/nginx-pipeline.conf input { beats { port => 5044; ssl => true } } filter { grok { match => { "message" => "%{COMBINEDAPACHELOG}" } } date { match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]; remove_field => ["timestamp"] } geoip { source => "clientip"; target => "geoip" } mutate { convert => { "response" => "integer" }; rename => { "clientip" => "source.ip" } } } output { # Conditional routing to different indices if [response] >= 500 { elasticsearch { hosts => ["https://es:9200"]; index => "errors-%{+YYYY.MM.dd}"; api_key => "${ES_KEY}" } } else { elasticsearch { hosts => ["https://es:9200"]; index => "nginx-%{+YYYY.MM.dd}"; api_key => "${ES_KEY}" } } }
Ingest Pipeline vs Logstash
FeatureIngest PipelineLogstash
LocationInside ElasticsearchSeparate process/server
Config formatJSON via API or Kibana UI.conf DSL files
Conditional routingLimited (reroute processor)Full if/else between outputs
Fan-outNo — single ES outputYes — multiple outputs simultaneously
Best forSimple enrichment and field extractionComplex routing, multi-source, fan-out

Data Streams
Create and Configure Data Streams
ElasticsearchCore Exam
✓
▾

A data stream stores time-series data across multiple backing indices transparently. It is the recommended approach for logs, metrics, and traces in Elastic Observability. Documents are always appended to the current write index.

🎯
Exam objective: Know data stream naming convention ({type}-{dataset}-{namespace}), how to create them, and their key constraints.
Data Stream API
# Naming: {type}-{dataset}-{namespace} logs-nginx.access-production # nginx access logs, production env metrics-system.cpu-staging # system cpu metrics, staging env traces-apm.spans-default # APM spans, default namespace # Create (requires a matching index template with data_stream:{}) PUT _data_stream/logs-nginx.access-production # Index a document (always append-only, @timestamp required) POST logs-nginx.access-production/_doc { "@timestamp": "2026-02-27T10:30:00Z", "message": "..." } # View details (backing indices, write index) GET _data_stream/logs-nginx.access-production # Manual rollover — creates a new backing write index POST logs-nginx.access-production/_rollover
Data Stream vs Regular Index
FeatureData StreamRegular Index
Best forTime-series: logs, metrics, tracesNon-time-series: products, users, config
WriteAppend-only to current write indexCreate, update, delete any document
@timestampRequired in every documentOptional
Backing storageMultiple hidden .ds-* indicesSingle index
ILMNative automatic rolloverManual ILM attachment
⚠
Documents in data streams cannot be updated directly. Use update_by_query or delete_by_query for modifications.

Index Templates
Create and Configure Index Templates
ElasticsearchCore Exam
✓
▾

An index template automatically applies settings, mappings, and aliases when a new index is created matching its pattern. For data streams, a matching template with "data_stream": {} is required before the stream can be created.

Index Template API
PUT _index_template/logs-nginx-template { "index_patterns": ["logs-nginx.*"], // matches by wildcard pattern "data_stream": {}, // enables data stream creation "priority": 200, // higher priority wins on conflicts "composed_of": ["logs-ecs-mappings", "logs-default-settings"], "template": { "settings": { "number_of_shards": 1, "number_of_replicas": 1, "index.lifecycle.name": "logs-default-policy" }, "mappings": { "properties": { "@timestamp": { "type": "date" }, "source.ip": { "type": "ip" }, "http.response.status_code": { "type": "integer" }, "message": { "type": "text" } } } } }
◆ Exam Question
You want to create a data stream named logs-myapp.errors-production. What must already exist in Elasticsearch?
A
An ILM policy — the data stream won't be created without one
B
An index template with index_patterns that matches logs-myapp.errors-production, including "data_stream": {}
C
A rollover alias pointing to an existing index
D
A backing index manually created as .ds-logs-myapp.errors-production-000001

Component Templates
Use Component Templates for Reusable Configuration
Elasticsearch
✓
▾

Component templates are reusable snippets of settings, mappings, or aliases composed into index templates via composed_of. They prevent duplication across many index templates and are the DRY principle applied to Elasticsearch configuration.

Component Template Pattern
# 1. Create reusable mapping component PUT _component_template/logs-ecs-mappings { "template": { "mappings": { "properties": { "@timestamp": { "type": "date" }, "host.name": { "type": "keyword" }, "source.ip": { "type": "ip" }, "message": { "type": "text" }, "log.level": { "type": "keyword" } }}} } # 2. Create reusable settings component PUT _component_template/logs-default-settings { "template": { "settings": { "number_of_shards": 1, "number_of_replicas": 1, "index.lifecycle.name": "logs-default-policy" }} } # 3. Compose into multiple index templates (DRY) PUT _index_template/logs-nginx-template { "index_patterns": ["logs-nginx.*"], "data_stream": {}, "priority": 200, "composed_of": ["logs-ecs-mappings", "logs-default-settings"], // Only add nginx-specific fields here "template": { "mappings": { "properties": { "http.response.status_code": { "type": "integer" } }}} }
ℹ
Component templates in composed_of merge in order. The index template's own template section applies last and overrides earlier components. Elastic ships built-in components like logs@mappings and metrics@settings you can extend.
