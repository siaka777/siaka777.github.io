APM Integration
Edit and Configure the APM Integration
APMFleet
▾

The APM integration is managed through Kibana's Fleet/Integrations interface. It configures the APM Server (receiving endpoint for agents) as part of an Elastic Agent policy. You can set the host/port, authentication (secret token or API keys), TLS, and RUM settings.

🎯
Exam objective: "Edit and configure the APM integration." You need to know how to navigate to it in Fleet, change the APM Server host/port, enable RUM, and set the secret token.
APM Integration Navigation & Key Settings
# Navigate to: Kibana → Fleet → Agent policies → [policy with APM] # OR: Kibana → Observability → APM → Add APM integration KEY SETTINGS IN THE APM INTEGRATION: Host: 0.0.0.0:8200 # APM Server listen address URL: https://apm-server:8200 # What agents connect to Secret token: mySecretToken # Shared secret for agent auth API key auth: enabled/disabled # Alternative to secret token Enable RUM: true # Allow browser agents (CORS) Allowed origins: ["*"] # CORS origins for RUM TLS: # Configure certificate/key for HTTPS enabled: true certificate: /path/to/cert.pem key: /path/to/key.pem Tail-based sampling: false # Advanced: sample based on full trace outcome
💡
RUM must be explicitly enabled in the APM integration. By default it's disabled. When enabled, the APM Server accepts data from browser-based JavaScript agents and you must also configure allowed CORS origins.

05
Traces & Spans
Understand Transactions, Spans & Distributed Tracing
APM
▾

APM data is organized into a hierarchy: a trace is the complete journey of a request across all services. Each service contributes a transaction (the entry point for that service). Within a transaction, individual operations are captured as spans (DB queries, HTTP calls, cache operations).

APM Terminology
TermDefinitionExample
TraceComplete end-to-end journey of a single request across all servicesUser clicks "Buy" → checkout → payment → inventory → email
TransactionEntry point for a single service — the top-level operationHTTP GET /api/checkout in the checkout-service
SpanA timed operation within a transactionSELECT * FROM orders — 12ms DB query
Trace IDUnique ID shared across all services for one requestabc123def456… (propagated via HTTP headers)
Parent IDLinks spans/transactions to their parentcheckout span → parent is the API gateway transaction
Service MapTopology view of service dependencies derived from trace dataShows checkout-api → payment-service → stripe (external)
Error groupClustered collection of similar errors across transactionsNullPointerException in OrderService.java:142
Distributed Trace Propagation
# How trace context is propagated between services # W3C Trace Context (standard format used by Elastic APM) # HTTP headers added by APM agent automatically: traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01 # version-traceId(32hex)-parentId(16hex)-flags tracestate: es=s:1 # vendor-specific state (es = Elastic, s = sample rate) # When service-B receives this, its APM agent: # 1. Reads the traceparent header # 2. Creates a new transaction linked to the same trace ID # 3. Sets its parent to the upstream span/transaction ID # → All services appear in the SAME waterfall in the APM app
💡
In the APM app, click any transaction → "Trace sample" → you see the full distributed waterfall. Click any span to see its details: DB statement, HTTP URL, error details, etc. This is what the exam will ask you to navigate.

APM Alerting
Create APM Alerts for Latency, Errors & Anomalies
AlertingKibana
▾

The APM app integrates with Kibana Alerting to fire notifications when service performance degrades. You can alert on transaction latency, error rate, anomaly score, or transaction throughput — all scoped to a specific service, transaction type, and environment.

# Creating an APM alert Navigate to: Kibana → Observability → APM → [service] → Alerts → Create alert OR: Observability → Alerts → Manage Rules → Create rule → APM APM Rule Types: Latency threshold — avg/95th/99th latency > X ms Error count — errors > N per minute Error rate — error % > X% of transactions Transaction throughput — tpm < N (alert on traffic drops) Anomaly (ML) — anomaly score > N on latency/throughput Example — Latency alert: Service: checkout-api Type: request Environment: production Aggregation: 95th percentile Threshold: is above 2000ms Window: last 5 minutes Action: Slack connector → #apm-alerts

08
APM + Logs Correlation
Correlate APM Traces with Logs & Infrastructure
APMLogs
▾

One of Elastic's most powerful features is correlation between APM traces and logs. When your application logs include the trace.id and transaction.id fields (injected by the APM agent), you can jump directly from a trace to its associated log lines — and vice versa.

Enabling Log Correlation
1
Configure APM agent to inject trace IDs into logs
Most agents do this automatically when using a supported logging framework (Log4j, Logback, Winston, structlog, etc.).
2
Ensure logs are in ECS format
Logs must have trace.id, transaction.id, and service.name fields using ECS field names for correlation to work.
3
Ship logs with Filebeat (ECS-compatible output)
Use Filebeat with the appropriate module or with json.message_key to parse structured JSON logs.
4
In APM app — click any transaction → Logs tab
The APM app automatically queries logs using the trace.id from the selected transaction — no manual filtering needed.
# Example correlated log line (ECS format) { "@timestamp": "2026-02-27T10:23:45.123Z", "log.level": "ERROR", "message": "Payment failed: card_declined for order 9981", "service.name": "payment-service", "trace.id": "4bf92f3577b34da6a3ce929d0e0e4736", ← KEY FIELD "transaction.id": "00f067aa0ba902b7", ← KEY FIELD "host.name": "payment-01.prod" }
ℹ️
In the APM app, the Infrastructure tab on a service shows which hosts/containers are running that service — derived by correlating service.name across APM data and Metricbeat/Agent data. No extra config needed.
