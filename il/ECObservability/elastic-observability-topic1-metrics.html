02 Enable and Analyze Predefined ML Jobs for Metrics
MLKibana
✓ DONE ▾

The Metrics app ships with predefined Machine Learning jobs that detect anomalies in your infrastructure metrics automatically — no ML expertise required. You just click "Enable" and they start running against your data.

🎯
Exam objective: "Use the Metrics app to enable and analyze the predefined machine learning jobs." Know how to navigate to them, enable them, and interpret the anomaly results in the UI.

How to Enable Predefined ML Jobs
1
Open the Metrics Inventory or Explorer
Kibana → Observability → Infrastructure
2
Click "Enable anomaly detection"
Found on the host tile detail view, or via the ML settings in the Metrics app toolbar.
3
Select the jobs to enable
Pick from the list of predefined jobs for your data type (hosts, Kubernetes, Docker).
4
Wait for results
Jobs run on historical data first (datafeed), then continue in real-time. Anomaly scores appear on the inventory tiles.
5
Analyze anomalies
Tiles with anomalies show a warning indicator. Click a tile → "Anomaly detection" tab to see the detailed anomaly timeline.
💡
Anomalies are displayed directly on the Inventory tile as a small warning icon. Click into the host and switch to the Anomaly detection tab to see the anomaly score timeline alongside the raw metric values.
03 Configure Metricbeat to Collect Metrics
MetricbeatConfig
✓ DONE ▾

Metricbeat is a lightweight shipper that collects system and service metrics from hosts and ships them to Elasticsearch. It uses modules (system, docker, kubernetes, nginx, mysql…) and metricsets (sub-datasets within a module).

🎯
Exam tasks include: Configuring Metricbeat modules, setting up output to Elasticsearch, enabling and disabling specific metricsets, and configuring the collection interval.
Interactive Metricbeat Config Builder
METRICBEAT.YML GENERATOR INTERACTIVE
← Configure settings and click Generate
Key Metricbeat Commands
# Enable a module metricbeat modules enable docker # List enabled modules metricbeat modules list # Test configuration metricbeat test config -e # Test Elasticsearch connection metricbeat test output # Run setup (creates dashboards, ILM policy, index template) metricbeat setup -e # Start Metricbeat systemctl start metricbeat
⚠️
Always run metricbeat test config -e before starting — syntax errors in the YAML will prevent Metricbeat from launching and the error messages give you the exact line to fix.
04 Metricbeat Modules & Metricsets
Metricbeat
✓ DONE ▾

A module groups related metrics from a single technology (e.g. MySQL). Each module contains metricsets — specific datasets within that technology. You enable/disable metricsets to control what gets collected.

05 Configure Metricbeat for Stack Monitoring
MetricbeatKibana
✓ DONE ▾

Stack Monitoring uses Metricbeat to monitor the Elastic Stack itself — Elasticsearch nodes, Kibana instances, Logstash, and Beats. This is the recommended approach (replacing the older internal collection method).

🎯
Exam objective: "Configure Metricbeat to gather Stack Monitoring data." You need to know how to enable the elasticsearch, kibana, and logstash modules on Metricbeat and point them at your Stack components.
# Step 1: Enable the elasticsearch-xpack module on Metricbeat metricbeat modules enable elasticsearch-xpack # modules.d/elasticsearch-xpack.yml - module: elasticsearch xpack.enabled: true period: 10s hosts: - https://localhost:9200 username: "remote_monitoring_user" password: "YOUR_PASSWORD" # Step 2: Enable kibana-xpack module metricbeat modules enable kibana-xpack # modules.d/kibana-xpack.yml - module: kibana xpack.enabled: true period: 10s hosts: - "http://localhost:5601" # Step 3: Output Metricbeat data to the MONITORING cluster # (best practice: use a separate monitoring cluster) output.elasticsearch: hosts: ["https://monitoring-cluster:9200"]
ℹ️
Use the built-in remote_monitoring_user account for Stack Monitoring — it has exactly the privileges needed. Avoid using the elastic superuser for this purpose.
06 Use Elastic Agent for Metrics Collection
Elastic AgentFleet
✓ DONE ▾

Elastic Agent is the unified, single-agent approach that replaces running separate Beats. One Agent replaces Metricbeat + Filebeat + Heartbeat. It is managed centrally through Fleet in Kibana using integrations and Agent policies.

Elastic Agent vs Metricbeat
AspectMetricbeatElastic Agent
InstallationOne per data type (Beat)Single agent for everything
ManagementLocal config files (YAML)Centralized via Fleet in Kibana
UpdatesManual per hostRemote upgrade from Fleet
Config changesEdit YAML, restartPush policy from Kibana, no restart
Metrics integrationsModulesIntegrations (System, Docker, K8s…)
Preferred for new deploymentsLegacy/direct control✅ Recommended
Fleet-Managed Agent Workflow
1
Set up Fleet server
Kibana → Fleet → Settings → Add Fleet Server. Generates enrollment URL and token.
2
Create an Agent Policy
Kibana → Fleet → Agent policies → Create policy. A policy groups integrations together.
3
Add integrations to the policy
Click the policy → Add integration → e.g. System, Docker, Kubernetes. Configure period, metricsets.
4
Enroll the Elastic Agent on each host
Run the enrollment command from the Fleet UI: elastic-agent install --url=... --enrollment-token=...
5
Monitor agent status in Fleet
All enrolled agents appear in Kibana → Fleet → Agents. Green = healthy, sending data.
Practice Quiz
You need to change the collection period for the System integration on 50 hosts managed by Elastic Agent Fleet. What is the most efficient approach?
ASSH into each of the 50 hosts and edit the YAML config file
BEdit the Agent Policy in Kibana Fleet — the change automatically propagates to all enrolled agents
CReinstall Elastic Agent on all 50 hosts with the new configuration
DCreate a new Fleet Server with the updated period and re-enroll all agents
07 Metrics Explorer & Custom Visualizations
Kibana
✓ DONE ▾

The Metrics Explorer (within Observability → Infrastructure) lets you build custom time-series metric charts without needing to go to Lens or TSVB. It's designed specifically for infrastructure metrics with built-in field suggestions.

# Metrics Explorer features Metrics: Choose fields to plot — Pick any numeric field from Metricbeat/Agent indices — Choose aggregation: avg, max, min, sum, rate, cardinality Split by: Group by a keyword field — host.name → one line per host — cloud.region → one line per AWS region — kubernetes.pod.name → one line per pod Filter by: KQL filter to scope data — e.g. cloud.provider : "aws" Chart type: Line, Area, Bar Timerange: From the global time picker
💡
To get the rate of change (e.g. network bytes per second from a cumulative counter), use the Rate aggregation in Metrics Explorer — it automatically calculates the derivative for you. No Painless scripting needed.
08 Alerting on Metrics Data
KibanaAlerting
✓ DONE ▾

Kibana's Alerting framework lets you create rules that trigger actions (email, Slack, PagerDuty, webhook) when metric thresholds are crossed. The Metrics app has built-in rule types specifically for infrastructure metrics.

Metrics Alert Rule Types
Rule TypeTriggers whenExample use
Inventory (Metric threshold)A metric on a host/pod/container exceeds a thresholdCPU > 90% for 5 minutes on any host
Custom thresholdAny Elasticsearch query result meets a condition95th percentile of response_time > 500ms
Anomaly detection (ML)ML anomaly score exceeds a set levelAnomaly score > 75 on memory usage
# Creating a Metric Threshold Alert Navigate to: Kibana → Observability → Alerts → Manage Rules → Create Rule Rule type: Inventory (Metric threshold) Conditions: Metric: system.cpu.total.pct Aggregation: Average Threshold: is above 0.90 # 90% For the last: 5 minutes Group by: host.name # alert per-host Actions (on alert): Connector: Email / Slack / PagerDuty Message template: uses Mustache variables e.g. "CPU alert: {{context.group}} exceeded 90%"
