02
ES|QL
Query Data with ES|QL (Elasticsearch Query Language)
ES|QLKibana
▾

ES|QL is a pipe-based query language introduced in Elasticsearch 8.11. It allows you to process, transform, and aggregate data using a series of commands chained with pipes (|). Available in Kibana Discover, Dashboards, and the ES|QL editor.

ES|QL Syntax & Examples
# Basic structure: SOURCE | CMD1 | CMD2 | ... # Find top error sources FROM logs-* | WHERE log.level == "ERROR" | STATS count = COUNT(*) BY service.name | SORT count DESC | LIMIT 10 # Calculate p95 latency per service FROM traces-apm.* | WHERE transaction.type == "request" | STATS p95 = PERCENTILE(transaction.duration.us, 95), avg_ms = AVG(transaction.duration.us) / 1000, total = COUNT(*) BY service.name | SORT p95 DESC # Enrich with EVAL (calculated field) FROM logs-nginx.* | EVAL is_error = http.response.status_code >= 400 | STATS errors = COUNT(*) WHERE is_error BY url.path | SORT errors DESC | LIMIT 5
Core ES|QL Commands
CommandPurposeExample
FROMSource index patternFROM logs-* , metrics-*
WHEREFilter documents (like SQL WHERE)WHERE log.level == "ERROR"
STATS … BYAggregation with optional groupingSTATS count=COUNT(*) BY service.name
EVALAdd a computed columnEVAL ms = duration_us / 1000
SORTOrder resultsSORT count DESC
LIMITRestrict number of rowsLIMIT 10
KEEPSelect specific columnsKEEP @timestamp, service.name, log.level
DROPRemove columnsDROP message, agent.*
RENAMERename a columnRENAME duration_us AS latency_us
DISSECT / GROKParse text fields inlineGROK message "%{IP:src} %{WORD:method}"
ℹ
KQL vs ES|QL: Use KQL for quick filtering and exploration in Discover. Use ES|QL for transformations, aggregations, calculated fields, and when you need tabular/analytical results or want to build a new kind of visualization.

Create Visualizations with Kibana Lens
LensCore Exam

Kibana Lens is the drag-and-drop visualization editor. You drag fields onto axes, and Lens suggests chart types. It can be used from Dashboards → Create panel → Lens, or from Discover → Visualize field. Lens is the recommended visualization tool for most use cases.

Dashboards
Build and Navigate Kibana Dashboards
KibanaCore Exam
▾

Dashboards compose multiple visualizations, saved searches, and controls into a single view. You can filter all panels simultaneously, drill through to Discover, and link dashboards together for a layered observability experience.

Creating & Using Dashboards
# Dashboard workflow Navigate: Kibana → Dashboards → Create dashboard Add panels: Add from library ← reuse existing saved visualizations Create visualization ← Lens editor inline Add text/markdown ← documentation panels Add controls ← interactive filters (Options List, Range Slider) Add links ← navigate to other dashboards Key dashboard features: Time picker ← controls all panels (top right) KQL filter bar ← filters all panels simultaneously Pin filters ← keep filter across dashboard navigation Drill-down ← click a chart value → go to Discover with filter applied Download CSV ← export panel data Share ← share link / embed in iFrame / PDF/PNG report
💡
On the exam, you may need to navigate between dashboards and filter data. Know that clicking a chart value creates a filter, and that the KQL bar at the top filters all panels at once.

Machine Learning
Enable and Use Anomaly Detection ML Jobs
MLKibana
▾

Elastic ML provides unsupervised anomaly detection — it learns a baseline from your historical data and alerts when behavior deviates. Each Observability app has predefined ML jobs you can enable with one click. Custom jobs can be built in Machine Learning → Anomaly Detection.

Predefined ML Jobs Per App
AppJob IDDetects
Metricsmetrics_hosts_memory_usageUnusual memory usage spikes on hosts
Metricsmetrics_hosts_network_inUnexpected network inbound traffic
Metricsmetrics_k8s_pod_cpuKubernetes pod CPU anomalies
Logslogs_ui_analysisAnomalous log event rate (unusual spike or drop)
Logslogs_ui_categoriesRare or novel log message categories
APMapm_tx_metrics (auto)Transaction latency and throughput anomalies per service
Uptime(via Anomaly alert rule)Unusual uptime/response time patterns
Enabling Predefined ML Jobs
# Metrics ML jobs Navigate: Kibana → Observability → Metrics → Anomaly detection → Enable jobs (metrics_hosts_memory_usage, etc.) # Logs ML jobs Navigate: Kibana → Observability → Logs → Anomalies tab → Enable log rate anomaly detection # APM ML — enabled automatically when: Navigate: Kibana → Observability → APM → [service] → Anomalies → Enable anomaly detection for the service # View all ML jobs Navigate: Kibana → Machine Learning → Anomaly Detection → Jobs → See job status, bucket span, datafeed state # Key ML job states: opened ← job running, datafeed active closed ← job stopped (can be reopened) failed ← error state, check job messages
💡
ML anomaly scores range from 0–100. Scores above 75 are considered critical. The "critical" threshold can be configured in alert rules. Anomaly results appear as colored bands in timeline views across all Observability apps.

Cross-pillar Correlation
Correlate Logs, Metrics, Traces, and Uptime Data
KibanaCore Exam
▾

The real power of Elastic Observability is cross-pillar correlation — jumping from a metric spike to related logs to the APM trace that caused it, all linked by shared fields like host.name, service.name, and trace.id.

Correlation Pathways
FromToLinking field / method
APM TransactionLogstrace.id — APM app → Transaction → Logs tab
APM ServiceInfrastructureservice.name + host.name — APM app → Infrastructure tab
Metrics (host spike)Logs on that hosthost.name — Metrics Inventory → click host → Logs
Log errorAPM trace for that requesttrace.id in log → click trace link in Logs app
Uptime monitor downLogs from that serviceservice.name → Logs app with filter
Any eventTimeline viewUse Kibana Discover with shared time range across apps
Required Fields for Correlation
# For APM ↔ Logs correlation — logs MUST contain: trace.id ← injected by APM agent automatically transaction.id ← injected by APM agent automatically service.name ← must match APM service name exactly # For Metrics ↔ APM correlation: host.name ← same value in both Metricbeat and APM agent config service.name ← same value in both logs and APM # Infrastructure tab in APM — auto-populated when: container.id ← APM agent picks this up automatically in containers kubernetes.pod.name ← from k8s metadata enrichment
