02
Heartbeat
Configure Heartbeat for Active Monitoring
HeartbeatConfig
▾

Heartbeat is an Elastic Beat that actively probes endpoints at a scheduled interval. It supports three monitor types: HTTP (checks web services), TCP (raw port connectivity), and ICMP (ping). Results are sent to Elasticsearch and appear in the Uptime app.

Heartbeat Configuration
# heartbeat.yml — monitor definitions heartbeat.monitors: # HTTP monitor — checks a web endpoint - type: http id: production-homepage name: "Production Homepage" urls: ["https://www.example.com"] schedule: '@every 1m' check.response: status: [200] body.contains: ["Welcome"] tags: ["production", "web"] # TCP monitor — checks a port is open - type: tcp id: db-port-check name: "Postgres Port 5432" hosts: ["db-01.prod:5432"] schedule: '@every 30s' # ICMP monitor — ping check (requires root) - type: icmp id: gateway-ping name: "API Gateway Ping" hosts: ["10.0.1.1"] schedule: '@every 10s' output.elasticsearch: hosts: ["https://es-cluster:9200"] username: "heartbeat_writer" password: "${HEARTBEAT_PWD}"
Monitor Types
TypeProtocolWhat it checksKey config
httpHTTP/SResponse status, body content, response time, TLS certurls, check.response.status, check.response.body
tcpTCPPort is open, optional send/receive string matchinghosts (host:port), check.send, check.receive
icmpICMPHost responds to ping, round-trip timehosts (IP or hostname)
⚠
ICMP requires root/elevated privileges to send raw ICMP packets. HTTP and TCP monitors do not.
Key Heartbeat Commands
heartbeat test config -c heartbeat.yml # validate config file heartbeat test output -c heartbeat.yml # test ES connectivity heartbeat run --once -c heartbeat.yml # run once (testing) heartbeat setup -e # load templates + dashboards

Elastic Synthetics
Configure Elastic Synthetics for Browser Testing
SyntheticsConfig
▾

Elastic Synthetics uses a real Chromium browser (via Playwright) to run scripted user journeys — simulating login flows, form submissions, multi-page flows. This goes beyond simple ping checks to validate end-to-end user experience.

⚡
Exam focus: Know the difference between Heartbeat (active ping/HTTP) and Elastic Synthetics (browser-based scripted journeys). Know how to enable and configure them in the Synthetics app UI.

// Elastic Synthetics browser journey script import { journey, step, expect } from '@elastic/synthetics'; journey('Login and view dashboard', ({ page }) => { step('Load login page', async () => { await page.goto('https://app.example.com/login'); await expect(page).toHaveTitle(/Login/); }); step('Enter credentials', async () => { await page.fill('[name=username]', 'testuser'); await page.fill('[name=password]', '${SYNTHETICS_PWD}'); await page.click('button[type=submit]'); }); step('Verify dashboard loads', async () => { await page.waitForSelector('.dashboard-widget'); await expect(page.locator('h1')).toHaveText('Welcome back'); }); });

Advanced Config
TLS Validation and Advanced HTTP Monitor Options
HeartbeatCore Exam
▾

Beyond basic YAML config, Heartbeat supports powerful per-monitor features: response validation, TLS settings, authentication, custom headers, and proxy configuration.

Advanced HTTP Monitor Config
- type: http id: api-health name: "API Health Check" urls: ["https://api.example.com/health"] schedule: '@every 30s' check.request: method: POST headers: 'Authorization': 'Bearer ${HEALTH_TOKEN}' 'Content-Type': 'application/json' body: '{"check": "full"}' check.response: status: [200, 204] body.contains: ["\"status\":\"ok\""] # TLS — for internal CA or custom certs ssl: certificate_authorities: ["/etc/ssl/ca.crt"] verification_mode: full # full / strict / certificate / none timeout: 15s

Your HTTP monitor fails with an SSL error because the endpoint uses a self-signed certificate from an internal CA. What should you configure to fix this without disabling TLS?
ASet ssl.verification_mode: none — disables all TLS validation

Ping Data
Understand Heartbeat Ping Documents & Timeline
KibanaHeartbeat
▾

Each time Heartbeat probes an endpoint, it writes a ping document to Elasticsearch. The Uptime app visualizes these over time. On a monitor's detail page you can see the availability graph, ping results list, duration trends, and — for HTTP monitors — full response details and TLS cert info.

Key Fields in Ping Documents
# Sample Heartbeat ping document { "@timestamp": "2026-02-27T10:30:00Z", "monitor.id": "production-homepage", "monitor.name": "Production Homepage", "monitor.type": "http", "monitor.status": "up", ← "up" or "down" "monitor.duration.us": 142000, ← microseconds "url.full": "https://www.example.com", "http.response.status_code": 200, "tls.certificate.not_after": "2026-06-15T00:00:00Z", "observer.geo.name": "us-east-1" ← probe location }
ℹ
Index patterns: heartbeat-* (standalone Heartbeat) or logs-uptime.*-* (Fleet/Agent-managed). The Uptime app handles both automatically.
06
Uptime Alerting
Create Uptime Alerts for Monitor Status & TLS Expiry
AlertingKibana
▾

Uptime alerting integrates with Kibana Alerting to fire when a monitor goes down or when a TLS certificate is approaching expiry. Alerts can notify via Slack, PagerDuty, email, webhook, and other connectors.

# Uptime rule types Navigate: Kibana → Observability → Uptime → Alerts → Create alert RULE TYPE 1: Monitor Status Fires when monitors report DOWN Config: consecutive checks, time window Filter by: monitor ID, tag, location Example: "Alert if any monitor tagged 'production' is DOWN for 3 consecutive checks" RULE TYPE 2: TLS Certificate Fires when a cert is approaching expiry Config: threshold (days until expiry) Example: "Alert if TLS cert expires in < 30 days" Applies to ALL monitors — no per-monitor config # Action/notification example Connector: PagerDuty Message: "Monitor {{context.monitorName}} is DOWN. URL: {{context.monitorUrl}}"
Practice Quiz
◆ EXAM QUESTION
You need a PagerDuty alert when any monitor tagged "critical" goes DOWN for 2 consecutive checks. Which rule type and filter configuration is correct?
ATLS Certificate rule — filter by "critical" tag, set threshold to 2 days
BMonitor Status rule — no filtering needed, PagerDuty fires for all monitors
CCreate a Watcher — Uptime doesn't natively support PagerDuty
DMonitor Status rule — filter by tag "critical", set 2 consecutive checks, PagerDuty action connector
