II
Snapshots
Configure Snapshot Repositories and Policies
ElasticsearchCore Exam
v

Snapshots are point-in-time backups of indices or data streams stored in an external snapshot repository (S3, GCS, Azure Blob, HDFS, or shared filesystem). They are also used for searchable snapshots in the Cold/Frozen ILM phases.

◆
Exam objective: Register a repository, create a snapshot, restore data from a snapshot. Know how SLM (Snapshot Lifecycle Management) automates this.
Snapshot Repository & SLM
# 1. Register a snapshot repository PUT _snapshot/my-s3-repo { "type": "s3", "settings": { "bucket": "my-elastic-backups", "region": "us-east-1", "base_path": "elasticsearch/snapshots" } } # 2. Create a manual snapshot PUT _snapshot/my-s3-repo/snapshot-2026-02-27 { "indices": "logs-nginx.*,metrics-system.*", "include_global_state": false } # 3. Monitor snapshot status GET _snapshot/my-s3-repo/snapshot-2026-02-27 GET _snapshot/my-s3-repo/_current ← in-progress snapshot # 4. Restore from snapshot POST _snapshot/my-s3-repo/snapshot-2026-02-27/_restore { "indices": "logs-nginx.*", "rename_pattern": "logs-nginx-(.*)", "rename_replacement": "restored-logs-nginx-$1" }
SLM — Snapshot Lifecycle Management
# Create an SLM policy for automated daily snapshots PUT _slm/policy/daily-snapshots { "schedule": "0 30 1 * * ?", ← cron: every day at 01:30 "name": "<daily-snap-{now/d}>", "repository": "my-s3-repo", "config": { "indices": ["logs-*", "metrics-*"], "include_global_state": false }, "retention": { "expire_after": "30d", "min_count": 5, "max_count": 50 } } # Trigger SLM policy manually POST _slm/policy/daily-snapshots/_execute # Navigate in Kibana: Stack Management → Snapshot & Restore → Policies

Security
Configure Roles, Users, and API Keys for Security
SecurityCore Exam
v

Elastic Security uses a role-based access control (RBAC) model. Roles define what a user can do (cluster privileges) and what data they can access (index privileges, field-level, document-level). API keys are preferred over passwords for programmatic access.

◆
Exam focus: Create a role with specific index privileges. Create a user assigned to that role. Create an API key with restricted privileges. Know built-in roles like kibana_admin, monitoring_user, apm_user.

IV
Stack Monitoring
Monitor the Elastic Stack Itself
ElasticsearchMonitoring
v

Stack Monitoring collects health and performance metrics from Elasticsearch, Kibana, Logstash, and Beats and displays them in Kibana → Stack Monitoring. Use a separate monitoring cluster in production to avoid noisy-neighbour effects.

◆
Key distinction: Metricbeat/Filebeat-based monitoring (recommended) vs internal monitoring. For Beats monitoring, set xpack.monitoring.enabled: true in the Beat config. For the monitoring cluster, ensure the correct Elasticsearch output.
Enabling Stack Monitoring
# Method 1: Metricbeat-based (RECOMMENDED) # On each Elasticsearch node — in metricbeat.yml: metricbeat.modules: - module: elasticsearch xpack.enabled: true ← routes to .monitoring-es-* indices period: 10s hosts: ["https://localhost:9200"] username: "remote_monitoring_user" password: "${MONITORING_PASS}" output.elasticsearch: hosts: ["https://monitoring-cluster:9200"] ← dedicated monitoring cluster # For Logstash monitoring — in logstash.yml: xpack.monitoring.enabled: true xpack.monitoring.elasticsearch.hosts: ["https://monitoring-cluster:9200"] # For Beats (Filebeat/Metricbeat) — in filebeat.yml: monitoring.enabled: true ← enables Beat self-monitoring # OR use xpack.enabled: true in the beat module for sending to monitoring cluster
⚠
Critical: xpack.enabled: true in a Metricbeat Elasticsearch/Logstash module causes data to be routed to the monitoring index pattern (.monitoring-*) instead of normal indices. This is what the Stack Monitoring app reads from. Forgetting this is a common mistake.

Index Management
Manage Indices: Mappings, Settings, and Aliases
ElasticsearchCore Exam
v

Beyond data streams, you need to manage the underlying indices — updating mappings, adjusting settings, creating aliases for rollover, and controlling shard allocation. The Index Management app in Kibana (Stack Management → Index Management) provides a UI for all of this.

Essential Index APIs
# View index settings and mappings GET logs-nginx-000001/_settings GET logs-nginx-000001/_mapping # Update mappings (can add new fields, can't change existing type) PUT logs-nginx-000001/_mapping { "properties": { "custom.field": { "type": "keyword" } } } # Update dynamic index settings PUT logs-nginx-000001/_settings { "number_of_replicas": 0 } ← can change replicas live # Static settings (number_of_shards) require index close/open POST logs-nginx-000001/_close PUT logs-nginx-000001/_settings ← apply static settings POST logs-nginx-000001/_open # Aliases POST _aliases { "actions": [ { "add": { "index": "logs-nginx-000001", "alias": "logs-nginx", "is_write_index": true } } ]} # Reindex — copy data from one index to another POST _reindex { "source": { "index": "old-logs" }, "dest": { "index": "logs-new-000001", "pipeline": "parse-nginx" } } # Force merge (reduce segment count — typically in warm phase) POST logs-nginx-000001/_forcemerge?max_num_segments=1
Dynamic vs Static Settings
SettingTypeCan change live?
number_of_replicasDynamicYes — PUT _settings anytime
index.lifecycle.nameDynamicYes — assign/change ILM policy live
index.routing.allocation.*DynamicYes — move shards to different tiers
number_of_shardsStaticNo — must close index first (or reindex)
index.codecStaticNo — set at creation time only

Cross-Cluster Replication
Configure Cross-Cluster Replication (CCR)
ElasticsearchCCR
v

Cross-Cluster Replication (CCR) continuously replicates indices from a leader cluster to a follower cluster. Used for disaster recovery, geo-distribution, and local reads. Follower indices are read-only and automatically kept in sync.

CCR Setup
# 1. Configure remote cluster (on the FOLLOWER cluster) PUT _cluster/settings { "persistent": { "cluster.remote.leader-cluster.seeds": ["leader-node-1:9300", "leader-node-2:9300"] }} # 2. Create a follower index (on the FOLLOWER cluster) PUT /follower-logs-nginx/_ccr/follow { "remote_cluster": "leader-cluster", "leader_index": "logs-nginx-000001" } # 3. Auto-follow pattern — replicate new indices automatically PUT _ccr/auto_follow/logs-pattern { "remote_cluster": "leader-cluster", "leader_index_patterns": ["logs-nginx-*"], "follow_index_pattern": "{{leader_index}}" } # Navigate: Stack Management → Cross-Cluster Replication
FeatureCCR
DirectionLeader → Follower (one-way)
Follower modeRead-only
Use casesDR (disaster recovery), geo-local reads, data locality
RequiresPlatinum/Enterprise license, remote cluster configured
Cross-cluster searchDifferent feature — query across clusters without replication

Fleet & Agent Management
Manage Elastic Agents and Fleet Policies
FleetCore Exam
v

Fleet is the central management layer for Elastic Agents — it pushes agent policies containing integrations, data streams, and configuration to all enrolled agents simultaneously. Understand how to manage agents at scale.

◆
Exam focus: Know how to add an integration to a policy, enroll a new agent, check agent health, and update policy settings. Fleet replaces manual Beat configurations at scale.
Fleet Agent Lifecycle
1
Create Agent Policy
Kibana → Fleet → Agent policies → Create policy. Name it, set monitoring settings.
2
Add Integrations
Click policy → Add integration: System, Nginx, APM, Kubernetes, AWS, etc. Each integration adds data streams and ingest pipelines automatically.
3
Enroll an Agent
Fleet → Add agent → select policy → copy enrollment command. Run on the host: elastic-agent install --url=https://fleet-server:8220 --enrollment-token=TOKEN
4
Monitor Agent Health
Fleet → Agents tab → check status (Healthy / Unhealthy / Updating / Offline). Click agent for logs and diagnostics.
5
Update Policy
Edit policy → change integration settings → Save. Fleet automatically pushes the updated config to all enrolled agents within seconds.
Fleet vs Manual Beats
FeatureFleet + Elastic AgentManual Beats
ConfigurationCentral UI, pushed automaticallyPer-host config files (filebeat.yml, etc.)
UpdatesFleet pushes policy changes liveManual edit + restart on each host
Multiple inputsSingle agent, multiple integrationsMultiple Beat processes needed
Best forScale: 10s to 10,000s of hostsSimple setups, legacy, custom pipelines
