02.01

The Data Ingestion Pipeline

Before any detection or investigation can occur, data must flow from its source into Elasticsearch. Understanding how that data is collected, transformed, and stored is foundational for a SIEM analyst — especially when data is missing, malformed, or normalized incorrectly.

💻
SOURCE
OS · App · Network · Cloud · Identity
→
🔌
COLLECTOR
Elastic Agent · Beats · API · Syslog
→
⚙️
TRANSFORM
Ingest Pipeline · Logstash · Processor
→
🗄️
STORE
Elasticsearch Index (ECS-normalized)
→
🔍
ANALYZE
Kibana · Alerts · Hunting · Cases

Key exam concept: Data can be ingested via Elastic Agent (preferred), standalone Beats, Logstash pipelines, or direct API calls. The preferred modern approach is Elastic Agent + integrations managed by Fleet, which handles normalization to ECS automatically.

02.02

Collection Methods Compared

Knowing when to use each collection method is a common exam scenario. Use this comparison as your reference.

Method Best For Managed By ECS Auto-Norm Transformation
Elastic Agent Endpoints, cloud, most modern data sources Fleet (Kibana) Yes ✓ Via integrations
Filebeat Log files, syslog, specific modules Manual config files Yes (modules) Ingest pipelines
Winlogbeat Windows Event Logs exclusively Manual config files Yes ✓ Built-in mapping
Packetbeat Network protocol analysis (DNS, HTTP, TLS) Manual config files Yes ✓ Protocol parsers
Metricbeat System & service metrics Manual config files Yes ✓ Modules
Logstash Complex transformations, legacy data, fan-out Logstash config (.conf) Manual Full pipeline (filter)
Direct API Custom apps, serverless, Lambda Application code Manual Application-side

Watch out: Beats are still widely deployed but are considered legacy. The exam may present scenarios where you must identify the better approach. For new deployments, Elastic Agent + Fleet is always preferred unless you have a compelling reason (e.g., Logstash for complex multi-step transformations).

02.04

Ingest Pipelines & Processors

An Ingest Pipeline is a sequence of processors that transform documents before they are stored in Elasticsearch. They live inside Elasticsearch itself (not Logstash). They are used to parse, enrich, rename, remove, and add fields.

PUT _ingest/pipeline/my-security-pipeline
{ "description": "Normalize Windows auth events to ECS", "processors": [ { "grok": { "field": "message", "patterns": ["%{TIMESTAMP_ISO8601:@timestamp} %{GREEDYDATA:event.action}"] } }, { "rename": { "field": "winlog.event_data.SubjectUserName", "target_field": "user.name" } }, { "geoip": { "field": "source.ip", "target_field": "source.geo" } }, { "set": { "field": "event.category", "value": "authentication" } } ] }
// Processor Reference — Click to explore
grok Parse unstructured text using named regex patterns PARSE
rename Rename a field (e.g., normalize vendor field to ECS) MAP
geoip Add geo location data from an IP address field ENRICH
set Set a field to a static or dynamic value WRITE
remove Delete a field from the document CLEAN
convert Change a field's data type (string → integer, etc.) TYPE
date Parse a date string and set @timestamp TIME
script Run a Painless script for complex logic LOGIC
// grok processor Use case: Parse raw syslog or unstructured log lines into named fields. "grok": { "field": "message", "patterns": ["%{SYSLOGTIMESTAMP:syslog.timestamp} %{IPORHOST:source.ip}"] } Result: message field is parsed → syslog.timestamp and source.ip created. Exam tip: Grok uses Oniguruma regex. Common patterns: %{IP}, %{NUMBER}, %{WORD}, %{GREEDYDATA}.
02.05

Data Streams, Index Templates & ILM

Elastic Security uses data streams to manage time-series security event data. They provide automatic rollover and lifecycle management, keeping your cluster healthy and your storage costs predictable.

Data Stream Naming Convention
# Format: {type}-{dataset}-{namespace} logs-endpoint.alerts-default # Elastic Defend alerts logs-windows.sysmon_operational-default # Sysmon events logs-aws.cloudtrail-default # AWS CloudTrail metrics-system.cpu-default # CPU metrics # Backing indices (auto-created by data stream) .ds-logs-endpoint.alerts-default-2024.01.01-000001 .ds-logs-endpoint.alerts-default-2024.02.01-000002

The Index Lifecycle Management (ILM) policy moves data through phases automatically:

🔥
HOT
Active writes + fast reads. SSD storage. Recent data.
→
🌡️
WARM
Read-only, less frequent queries. Can use cheaper storage.
→
❄️
COLD
Rarely accessed. Object storage. Searchable but slow.
→
🧊
FROZEN
Long-term archive. Cached on demand. Cheapest tier.
→
🗑️
DELETE
Data purged per retention policy.

SIEM analyst relevance: Analysts must understand ILM because older events may be in COLD or FROZEN tiers, requiring async searches. If an alert references old data, it may not appear in standard searches — you need to expand the search tier in Discover settings.

02.06

ECS Normalization in Practice

ECS normalization is the process of mapping vendor-specific field names to the standard ECS field schema. This is what allows a detection rule written once to work across Windows events, Linux audit logs, and cloud logs simultaneously.

Before vs After ECS Normalization
// RAW Windows Security Event Log (before normalization) { "EventID": "4624", "SubjectUserName": "SYSTEM", "TargetUserName": "jdoe", "IpAddress": "192.168.1.50", "LogonType": "3" } // AFTER ECS normalization (stored in Elasticsearch) { "@timestamp": "2024-03-15T08:42:11.000Z", "event.category": ["authentication"], "event.type": ["start"], "event.outcome": "success", "user.name": "jdoe", "source.ip": "192.168.1.50", "winlog.logon.type": "Network", "winlog.event_id": 4624 }

Notice how the normalized document uses standard ECS fields (user.name, source.ip, event.outcome) while the vendor-specific fields (winlog.*) are preserved for extra context. This means a detection rule checking event.outcome: "failure" works across Windows, Linux, and cloud authentication logs.

Important ECS multi-value fields: event.category and event.type are arrays in ECS. A single event can belong to multiple categories (e.g., ["authentication", "session"]). Always check with is one of operators in KQL, not strict equality.

Topic 2 · Practice Questions
QUESTION 01
An analyst needs to ingest Palo Alto firewall logs that require complex multi-step transformations including conditional field logic that varies by log type. Which ingestion tool is MOST appropriate?
Correct: Logstash is the right choice for complex, conditional multi-step transformations. While Elastic Agent and ingest pipelines handle most cases, Logstash's full Ruby-based conditionals, multiple filter plugins, and fan-out routing capabilities make it the best tool for complex transformation logic that varies by log type.
QUESTION 02
In an ECS-normalized document, a Windows 4624 logon event sets event.category to ["authentication"]. An analyst writing a KQL query wants to match this. Which KQL syntax is correct?
Correct: event.category: "authentication" — KQL uses colon syntax for field matching, and even though event.category is an array field, KQL automatically searches within the array. You do NOT need to use array bracket syntax. Option C would also match but is not standard KQL syntax.
QUESTION 03
A new Elastic Agent policy needs to collect endpoint security events AND normalize them to ECS automatically. What must be added to the agent policy to enable this?
Correct: The Elastic Defend integration — Adding this integration to an agent policy enables EDR-level endpoint telemetry (process, file, network, registry events) and automatically normalizes everything to ECS. There is no need for Logstash, custom pipelines, or additional Beats.
QUESTION 04
An analyst investigates an alert but cannot find events older than 7 days in Discover. The cluster uses ILM. What is the most likely reason?
Correct: Cold/Frozen tier limitation. ILM moves older data to COLD and FROZEN tiers where data is on object storage. Standard searches don't automatically include these tiers. To search older data, analysts need to run async searches and ensure the search settings include cold/frozen tier data. Elasticsearch does NOT deduplicate events automatically.
QUESTION 05
Which ingest processor would you use to enrich a security event with geographic location information based on a source IP address?
Correct: geoip — The geoip processor looks up an IP address against the MaxMind GeoIP database bundled with Elasticsearch and adds geographic fields like source.geo.country_name, source.geo.city_name, and coordinates. This is used extensively in SIEM to enrich authentication and network events with location context.
QUESTION 06
What is the correct data stream naming convention for security logs collected by the Windows integration, in the "production" namespace?
Correct: logs-windows.security-production — Elastic data stream names follow the format {type}-{dataset}-{namespace}. Type is always logs, metrics, or traces. Dataset is integration_name.log_type. Namespace is user-defined (commonly "default" or environment name like "production").
