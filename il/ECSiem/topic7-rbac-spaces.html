07.01RBAC OVERVIEW

Elastic Security uses Role-Based Access Control (RBAC) to govern what users can see and do. Access is controlled at two layers: Kibana privileges (which features a user can access) and Elasticsearch index privileges (which data they can read or write). Both layers must be configured correctly.

Roles are assigned to users or user groups. A user can have multiple roles whose privileges are additive — Elastic does not support privilege denial (there is no "deny" — only grant or no-grant).

Two Privilege Layers
Kibana privileges
→ Control feature access in the UI
→ Examples: can view Alerts? Create rules? Manage cases?

Elasticsearch index privileges
→ Control which indices/data streams
→ Examples: read, write, manage, view_index_metadata
Privilege Model
Privileges are additive only.
Multiple roles = union of all permissions.
There is no DENY keyword.

To restrict access: simply
do not grant the privilege.

Least privilege principle: grant
only what is needed for the role.

Key exam concept: If a user needs to be prevented from seeing certain data, the correct approach is to not include that index pattern in their role's index privileges — not to add a deny rule. Elastic RBAC has no deny mechanism. Field-level security (FLS) and document-level security (DLS) can further restrict what within an index a user can see.

Elastic Security ships with four built-in roles designed for security use cases. These are the most exam-tested roles — know their names, what they can and cannot do.

Click a privilege level below to understand what it means in practice:

VIEWER
Read-only access to Elastic Security features — dashboards, alerts, cases, rules
Viewer role details:
• Can view alerts but cannot change their status (Open/Closed/Acknowledged)
• Can read cases but cannot add comments or change status
• Can view detection rules but cannot enable, disable, or modify them
• Cannot access Fleet, Endpoint management, or any write operations
• Typical user: executive dashboards, compliance auditors, read-only SOC oversight
TIER 1
Alert triage analyst — can work alerts and create/manage cases with limited scope
Tier 1 Analyst details:
• Can view AND update alert status (Open → Acknowledged → In Progress → Closed)
• Can create new Cases and add comments to own cases
• Cannot create or modify detection rules
• Cannot manage endpoints (no isolation, no response actions)
• Typical user: junior SOC analysts performing first-level alert triage
TIER 2
Senior analyst — full case management, read rules, limited endpoint visibility
Tier 2 Analyst details:
• Full alert triage plus bulk operations
• Full Cases management: create, assign, close, tag, attach timelines
• Can read (but NOT modify) detection rules — good for review without change risk
• Can view endpoint details but cannot perform response actions
• Typical user: senior SOC analysts, incident responders without admin access
TIER 3
Power analyst — full access including rule management and endpoint response
Tier 3 Analyst details:
• All Tier 2 privileges PLUS:
• Can create, edit, enable/disable detection rules
• Can configure rule actions (Slack, email, webhooks)
• Can perform endpoint response actions (isolate, kill process, run commands)
• Can manage exception lists and shared exception lists
• Typical user: senior detection engineers, IR team leads, SOC managers

Exam tip — Tier 2 read-only rules: The distinction between Tier 2 (read rules only) and Tier 3 (write rules) is frequently tested. A scenario might describe an analyst who needs to "review detection logic but not change it" — that's Tier 2. An analyst who "needs to create new detection rules" needs Tier 3.

// Spaces — Key isolation concepts What IS isolated per Space: Saved objects: • Detection rules → Rules in Space A don't appear in Space B • Cases & comments → Completely separate per Space • Timelines → Saved timelines don't cross Spaces • Dashboards & visualizations • Index patterns / data views • Saved searches • Connectors (Slack/JIRA) • Exception lists What is NOT isolated (shared across Spaces): • Raw data in Elasticsearch indices • Users and roles • Fleet agents and integrations • Machine learning jobs • Elasticsearch-level ILM policies

Exam trap: The underlying Elasticsearch data is NOT isolated by Spaces. Spaces only isolate Kibana saved objects (rules, cases, dashboards, etc.). If two Spaces both have index privileges to logs-endpoint*, analysts in both Spaces can query the same raw events. Space isolation is a Kibana UI concern, not an Elasticsearch data concern.

07.04INDEX PRIVILEGES

Elasticsearch index privileges control what users can do with specific indices or data streams. For Elastic Security, analysts typically need read and view_index_metadata on event indices, and specific privileges on the alerts index.

// Common Elasticsearch Index Privileges for Elastic Security read → Search and read documents in the index view_index_metadata → View index settings/mappings (needed for Discover filters) write → Index new documents (needed for alert status updates) manage → Manage index settings (admin only) monitor → View index stats and health create_index → Create new indices (admin only) delete → Delete documents (highly restricted) maintenance → Refresh, flush, force merge (ops/admin) // Minimum privileges for a read-only SOC analyst: { "index": [ { "names": ["logs-*", ".alerts-security.alerts-*"], "privileges": ["read", "view_index_metadata"] } ] } // Analyst who can update alert status also needs write on alerts index: { "names": [".alerts-security.alerts-*"], "privileges": ["read", "view_index_metadata", "write"] }
Field-Level Security (FLS)
Restricts which fields within an
index a user can see.

Example: hide user.password
or payment.card_number from
certain analyst roles.

Uses grant (include) or
except (exclude) field lists.
Document-Level Security (DLS)
Restricts which documents
(rows) a user can see using
a query filter.

Example: analysts can only
see events from hosts in
their assigned region:
{"term":{"host.region":"eu-west"}}

FLS + DLS exam scenarios: A scenario says "Analysts should only see events from their own team's endpoints" → use Document-Level Security with a term query on a host group field. "Analysts should not see the raw process command line arguments" → use Field-Level Security to exclude process.command_line. Knowing which restriction type applies to which scenario is key.

Trusted Applications

Critical exam distinction — Exceptions vs Trusted Applications:
Exception lists suppress specific alerts for specific conditions. They still allow the event to be ingested and indexed.
Trusted Applications tell Elastic Defend to completely ignore a specific application — no events are generated at all from that process.
Endpoint exceptions are the middle ground: pushed to the agent, prevent event generation for specific rule conditions.

ACCESS CONTROL QUIZ
QUESTION 01
A Tier 1 analyst needs to update alert statuses from Open to Acknowledged but cannot create or modify detection rules. Which built-in role satisfies BOTH requirements using least privilege?
Correct: Tier 1 Analyst. Tier 1 analysts can update alert status (Open → Acknowledged → Closed) and create Cases. They cannot create or modify detection rules, satisfying least privilege. The Viewer role cannot update status at all. Tier 2 could read rules (unnecessary here). Tier 3 grants full rule management — too much privilege for this use case.
An analyst in the SOC Space creates a set of custom detection rules. A colleague in the Cloud Security Space needs to use those same rules. What must the administrator do?
QUESTION 02
Correct: Export from SOC Space and import into Cloud Security Space. Detection rules are saved objects scoped to their Space. They are NOT automatically shared. The correct workflow is to export rules as an NDJSON file from the source Space (Security → Rules → Export) and import them into the target Space. Roles do not grant cross-space access to saved objects.
QUESTION 03
A company requires that SOC analysts can only see events from endpoints in their geographic region (e.g., region: "eu-west"). What Elasticsearch privilege feature enables this?
Correct: Document-Level Security (DLS). DLS restricts which documents (rows) a user can see within an index using a query filter. A DLS filter of {"term":{"host.region":"eu-west"}} on the analyst's role means they can only retrieve documents matching that filter. FLS restricts fields, not documents. Spaces isolate saved objects, not raw event data.
QUESTION 04
An analyst notices that when they try to change an alert's status from Open to Closed, they receive a permission error. They have the Viewer role. What is the minimum role change needed to fix this?
Correct: Upgrade to Tier 1 Analyst. The Viewer role is strictly read-only — it cannot update alert status. Tier 1 Analyst is the minimum role that includes alert status update capability. Adding raw index privileges alone (A/D) won't work because Kibana privilege checks happen at the application layer before index privileges. The Tier 1 Analyst role bundles both the Kibana feature privilege and the necessary index privilege. Superuser (C) violates least privilege.
QUESTION 05
What is the key difference between a "Trusted Application" and an "Endpoint Exception" in Elastic Defend?
Correct: Trusted Applications suppress all monitoring of a process; Endpoint Exceptions suppress specific rule conditions. A Trusted Application tells Elastic Defend: "never generate telemetry events for this signed/known process." An Endpoint Exception says: "when THIS specific rule condition is met AND these field values match, don't generate an alert." Trusted Apps are broader, Endpoint Exceptions are scoped. Neither makes them functionally identical.
QUESTION 06
A security administrator needs to ensure that analysts cannot see the raw process.command_line field in event data (it contains sensitive credentials passed via CLI). Which feature should be configured?
Correct: Field-Level Security (FLS). FLS restricts which fields within documents a user can see. Configuring FLS on the analyst's role to exclude process.command_line means they can still see the document (and all other fields) but that specific field is redacted/invisible. DLS (A) would hide the entire event document — much too broad. Spaces (C) don't restrict data. Ingest pipeline removal (D) would delete the data permanently for everyone, not just analysts.
QUESTION 07
In Elastic Security's RBAC model, if User A has Role 1 (which grants read on logs-endpoint*) and Role 2 (which grants write on .alerts-security.alerts-*), what are their effective privileges?
Correct: Privileges are additive — the user gets the union of all role permissions. Elastic RBAC uses a strictly additive model. Multiple roles are merged: all granted privileges from all assigned roles are combined. There is no role conflict, no precedence, and no deny mechanism. The user in this scenario has both read on endpoint logs AND write on the alerts index. This is why least privilege matters — once a privilege is granted via any role, it cannot be "taken back" by another role.
