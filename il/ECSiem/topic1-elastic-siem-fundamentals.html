01.01

What is SIEM?

Security Information and Event Management (SIEM) is a platform that aggregates, correlates, and analyzes security event data from across an organization's IT environment. It provides real-time threat detection, alerting, and compliance reporting from a single pane of glass.

SIEM combines two older disciplines: SIM (Security Information Management — long-term log storage and reporting) and SEM (Security Event Management — real-time monitoring and correlation).

🔍

Collect

Aggregate logs from endpoints, network devices, cloud, applications, and identity systems.

🔗

Correlate

Connect related events across sources using rules, ML, and behavioral analytics.

🚨

Alert

Generate actionable alerts when suspicious patterns match defined detection rules.

📊

Report

Produce compliance reports, dashboards, and investigation timelines for SOC teams.

🏃

Respond

Support incident response through cases, timelines, and orchestration capabilities.

01.02

Elastic Stack Architecture

Elastic Security is built on top of the Elastic Stack (ELK). Understanding the data flow is critical for every SIEM analyst — from ingestion all the way to detection.

Data Sources Endpoints · Network · Cloud · Apps
↓
Elastic Agent Single agent, all integrations
Beats Filebeat · Winlogbeat · Auditbeat
Logstash Transform & enrich
↓
Elasticsearch Store, index & search
↓
Kibana / Elastic Security SIEM UI · Detection · Hunting · Cases

Key exam point: Elastic Agent is the modern, preferred collection method. It replaces deploying multiple Beats individually. It is managed via Fleet in Kibana and supports hundreds of integrations.

01.03

Key Components Deep Dive

Elasticsearch is the distributed search and analytics engine at the core of the Elastic Stack. Key concepts for SIEM analysts:

# Core Elasticsearch concepts for SIEM Index → Named collection of documents (e.g., logs-endpoint.alerts-*) Document → Single JSON record representing a security event Shard → Horizontal partition for scalability Replica → Copy of shard for redundancy Mapping → Schema defining field types (keyword, date, ip, etc.) ILM → Index Lifecycle Management — hot/warm/cold/frozen/delete

For SIEM, Elasticsearch stores all security events. Analysts query it via KQL, EQL, or ES|QL from the Kibana UI — no need to write raw Elasticsearch queries directly.

Kibana is the visualization and management layer. For SIEM analysts, the critical sections are:

🛡️

Security App

Main SIEM interface: Overview, Alerts, Timelines, Cases, and more.

🔎

Discover

Ad-hoc log search using KQL. The analyst's primary investigation starting point.

📈

Dashboards

Pre-built and custom security dashboards for monitoring and reporting.

⚙️

Stack Management

Index patterns, ILM policies, users, roles, and spaces administration.

Elastic Agent is a unified, multi-purpose data collection agent. It replaces separate Beats deployments and provides:

# Elastic Agent capabilities Data Collection → Logs, metrics, traces from any source Endpoint Security → EDR functionality via Elastic Defend integration Network Monitoring → Packet capture and flow analysis Fleet Managed → Centrally managed via Kibana Fleet UI Policy Based → Agent policies define what data to collect Tamper Protected → Can be locked to prevent unauthorized changes

Exam tip: Know the difference between Elastic Agent, individual Beats (Filebeat, Winlogbeat, Packetbeat), and Logstash. Elastic Agent is preferred; Logstash is used for complex transformation pipelines.

Fleet is the centralized management system for Elastic Agents. Accessible from Kibana → Management → Fleet.

# Fleet concepts Fleet Server → Central server agents connect to for management Agent Policy → Defines integrations, data collection config, and outputs Integration → Pre-built connector for a specific data source Enrollment Token → Used to register a new agent with Fleet Agent Status → Healthy / Unhealthy / Updating / Offline / Unenrolled Output → Where agents send data (usually Elasticsearch or Logstash)

From Fleet, analysts and admins can remotely push policy updates, upgrade agents, unenroll agents, and view agent health — all without SSH access to endpoints.

01.04

Elastic Common Schema (ECS)

ECS defines a standard set of field names and data types across all Elastic security data. It enables consistent correlation across different data sources without custom field mappings.

Think of ECS as the "universal language" that allows a Windows event to be correlated with a firewall log and a DNS query — because they all use the same field names.

# ECS field categories used in SIEM @timestamp → Event time (ISO 8601) event.* → category, type, action, outcome, severity host.* → hostname, ip, os.name, os.version user.* → name, id, domain, email process.* → name, pid, parent.pid, executable, args network.* → transport, protocol, direction source.* → ip, port, geo.country_name destination.* → ip, port, geo.country_name file.* → name, path, hash.sha256, size dns.* → question.name, answers.data, type threat.* → tactic.id, technique.id (MITRE ATT&CK)

Why ECS matters for the exam: Detection rules, timelines, and dashboards all rely on ECS-normalized fields. Understanding field names like process.name, user.name, and destination.ip is essential for writing queries and building detection logic.

01.05

Kibana Spaces & Deployment Models

Spaces in Kibana are isolated environments within a single Elastic cluster. Each space has its own dashboards, saved searches, detection rules, and cases. Users can be granted access to specific spaces based on roles.

🏠

Default Space

Exists by default. Can be renamed. All objects created here unless another space is selected.

🏢

Custom Spaces

Created for different teams, business units, or security domains (e.g., "SOC Team", "Cloud Security").

🔒

Space Isolation

Detection rules, cases, and saved objects are per-space. Data indices are cluster-wide.

☁️

Elastic Cloud vs Self-Managed

Elastic Cloud is SaaS (hosted by Elastic). Self-managed runs on-prem or your own cloud VMs.

Exam tip: RBAC in Elastic is applied via Roles and Role Mappings. A role can grant access to specific Kibana features within specific Spaces, and control index-level privileges in Elasticsearch. This is covered in depth in Topic 7.

Topic 1 · Practice Questions

QUESTION 01
Which component of the Elastic Stack serves as the primary storage and search engine for security event data?
Correct: Elasticsearch is the distributed search and analytics engine where all security events are stored and indexed. Kibana is the UI layer, Logstash transforms/enriches data before indexing, and Elastic Agent collects data from endpoints.
QUESTION 02
A security analyst wants to deploy a single agent on endpoints that handles log collection, metrics, and EDR. Which Elastic product should they deploy?
Correct: Elastic Agent with Elastic Defend integration. Elastic Agent is the modern unified agent that replaces multiple Beats. Adding the Elastic Defend integration enables EDR capabilities (endpoint protection, process monitoring, file integrity, etc.) alongside log and metrics collection.
QUESTION 03
In Elastic Common Schema (ECS), which field would you use to identify the name of a process that triggered a security event?
Correct: process.name is the ECS-standard field for the process name. ECS uses dot-notation namespaces: process.* covers all process-related fields including process.pid, process.parent.pid, process.executable, etc.
QUESTION 04
A SOC manager wants to give the "Cloud Security" team access to their own dashboards and detection rules without exposing data from the "Windows SOC" team. What Kibana feature enables this isolation?
Correct: Kibana Spaces with role-based access. Spaces provide logical isolation of saved objects (dashboards, detection rules, cases, etc.) within a single cluster. Roles control which users can access which spaces, providing team isolation without needing separate infrastructure.
QUESTION 05
What is the primary purpose of Fleet in Elastic Security?
Correct: Fleet centrally manages Elastic Agent deployments. From Fleet in Kibana, you can enroll new agents, assign agent policies (which define what integrations/data to collect), push policy updates, upgrade agent versions, and monitor agent health — all from a single UI without direct access to endpoints.
