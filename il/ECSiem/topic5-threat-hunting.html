05.01

What Is Threat Hunting?

Threat hunting is a proactive, human-led investigation process that searches for threats that have evaded automated detection. Unlike alert triage (reactive), threat hunting starts with a hypothesis derived from threat intelligence, MITRE ATT&CK techniques, or analyst intuition — then actively looks for evidence in the data.

In Elastic Security, threat hunters primarily work in Discover, Timelines, and the Osquery Manager, using KQL, EQL, and ES|QL to interrogate raw event data across any time range.

Reactive (Alert Triage)
Starts from an alert.
Rule fired → analyst investigates.
Alert-driven, rule-dependent.
Catches known patterns only.

Limitation: only finds what
rules are configured to catch.
Proactive (Threat Hunting)
Starts from a hypothesis.
Analyst queries raw data directly.
Not dependent on existing rules.
Can find unknown/novel threats.

Outcome: new IOCs, new rules,
hardened detections.

The hunt lifecycle:
1. Form hypothesis — from threat intel, ATT&CK, news, intuition
2. Collect & search — query raw events with KQL/EQL/ES|QL
3. Analyze results — look for anomalies, patterns, outliers
4. Respond or document — create rules, cases, or close as benign
5. Improve detections — turn findings into new detection rules

05.02

The Three Query Languages

Elastic Security supports three query languages for threat hunting. Understanding the strengths, syntax differences, and appropriate use cases for each is heavily tested on the exam.

KQL
Kibana Query Language
Simple, human-readable field matching. Used in Discover, alert filters, and most SIEM UI search bars.
Discover searches Alert filtering Dashboard filters Custom Query rules Timeline queries
EQL
Event Query Language
Sequence detection and event correlation. Requires event category prefix. Used for attack chains and process trees.
Sequence detection Process trees Attack chain hunting Event Correlation rules
ES|QL
Elasticsearch Query Language
Pipe-based analytics language for aggregation, transformation, and enrichment. Like SQL + pipes. New and powerful.
Aggregations Top-N analysis Statistical hunting Data transformation ES|QL rules

Critical syntax differences — exam trap:
KQL uses field: "value" (colon). EQL uses field == "value" (double equals). ES|QL uses WHERE field == "value" (SQL-like). Mixing these up is the most common exam mistake. KQL uses AND/OR/NOT. EQL uses and/or/not. ES|QL uses AND/OR/NOT in WHERE clauses.

05.04

KQL Syntax Deep Dive

KQL (Kibana Query Language) is the most-used query language in Elastic Security. It is used in Discover, alert filtering, Timeline, and as the query language for Custom Query detection rules.

// ─── KQL SYNTAX REFERENCE ─────────────────────────────────────────── // Field matching (most common) process.name: "powershell.exe" event.category: "authentication" // Wildcard process.args: *EncodedCommand* // * wildcard, no quotes needed destination.domain: *.onion // Boolean operators (UPPERCASE) process.name: "cmd.exe" AND user.name: "Administrator" event.outcome: "failure" OR event.outcome: "unknown" NOT source.ip: "10.0.0.0/8" // Multi-value (parentheses) process.name: ("cmd.exe" OR "powershell.exe" OR "wscript.exe") // Range queries destination.port: [1024 TO 65535] event.risk_score: >70 @timestamp: ["2024-03-01" TO "2024-03-31"] // Exists check process.hash.sha256: * // field has any value // Nested fields process.parent.name: "WINWORD.EXE" source.geo.country_iso_code: ("RU" OR "CN" OR "KP")

Common KQL mistakes: (1) Using == instead of : — that's EQL syntax. (2) Using lowercase and/or — KQL requires uppercase AND/OR/NOT. (3) Quoting wildcards: write process.args: *enc* not process.args: "*enc*" — quotes disable wildcards.

05.05

ES|QL — The Analytics Powerhouse

ES|QL (Elasticsearch Query Language) is a pipe-based analytics language introduced to replace complex aggregation queries. It is ideal for statistical threat hunting — finding outliers, calculating baselines, and identifying anomalous patterns at scale.

// ─── ES|QL SYNTAX REFERENCE ────────────────────────────────────────── // Format: SOURCE | COMMAND1 | COMMAND2 | ... // Basic structure FROM logs-endpoint* | WHERE event.category == "process" | WHERE @timestamp > NOW() - "24h" | STATS count = COUNT(*) BY process.name, host.name | SORT count DESC | LIMIT 20 // Find rare processes (low host count = suspicious) FROM logs-endpoint* | WHERE event.type == "start" | STATS host_count = COUNT_DISTINCT(host.name) BY process.executable | WHERE host_count == 1 // only seen on 1 host | SORT host_count ASC // Key ES|QL commands FROM → data source (index pattern) WHERE → filter rows STATS ... BY → aggregate (like GROUP BY in SQL) SORT → order results LIMIT → max rows returned EVAL → compute new fields KEEP → select specific fields to display DROP → remove fields from output RENAME → rename a field in output DISSECT → parse text using patterns GROK → extract fields using regex patterns

Exam tip — KQL vs ES|QL choice: If the scenario asks to find specific events matching conditions → use KQL. If the scenario asks to count, aggregate, rank, or identify statistical outliers → use ES|QL. ES|QL cannot detect event sequences (use EQL for that). ES|QL is accessed in Discover by switching the query language selector.

05.07

Timelines & Osquery

Timelines are the analyst's investigation notebook in Elastic Security. During threat hunting, analysts use Timelines to collect, organize, and annotate events that support a hunt finding. They support KQL, EQL, and ES|QL queries directly.

Timeline Features
Pinned events — bookmark key events
Notes — analyst annotations per event
Query bar — KQL/EQL/ES|QL search
Time range — custom date/time window
Templates — pre-built hunt templates
Attach to Case — link to incident
Saved Timelines — persist for team sharing
Osquery Manager
Live endpoint interrogation via SQL.
Requires Osquery integration deployed.

SELECT * FROM processes
WHERE name = 'mimikatz.exe'

Queries run live on the host,
not on historical indexed data.
Great for volatile data: processes,
network connections, users.
// Osquery — Live endpoint queries during a hunt // All running processes and their hashes SELECT pid, name, path, md5, sha256 FROM processes JOIN hash ON processes.path = hash.path; // Active network connections SELECT pid, p.name, local_address, remote_address, remote_port, state FROM process_open_sockets AS s JOIN processes AS p USING (pid) WHERE state = 'ESTABLISHED'; // Persistence — Scheduled tasks (Windows) SELECT name, action, path, enabled FROM scheduled_tasks WHERE enabled = 1;

Timelines vs Discover: Both use KQL/EQL queries. Timelines are designed for investigation documentation — they let you pin events, write notes, and attach findings to Cases. Discover is better for ad-hoc exploration and data overview. For a threat hunt that needs to be shared with the team or escalated, always build a Timeline.

QUESTION 01
A threat hunter wants to find processes that ran on only ONE host across the entire environment in the past 7 days — a classic "rare process" hunt. Which query language is BEST suited for this task?
Correct: ES|QL with COUNT_DISTINCT. This is a statistical aggregation task — counting how many unique hosts have seen each process executable. KQL cannot aggregate; it only filters. EQL sequences don't apply here. ES|QL with STATS host_count = COUNT_DISTINCT(host.name) BY process.executable | WHERE host_count == 1 is the precise tool for this.
QUESTION 02
Review this query. What is WRONG with it?
process.name == "powershell.exe" and process.args: *EncodedCommand*
Correct: KQL uses colon (:) not double-equals (==) for field matching. The query mixes EQL syntax (==) with KQL syntax (:). In KQL, equality is written as process.name: "powershell.exe". The == operator is EQL syntax only. The wildcard on process.args is valid KQL. AND (uppercase) is correct for KQL.
QUESTION 03
A hunter wants to detect when: (1) a user runs wscript.exe, (2) which spawns cmd.exe, (3) which then makes a network connection — all within 3 minutes on the same host. Which query type achieves this?
Correct: EQL sequence query. Only EQL can enforce ordered event sequences within a time window on the same entity. The query would be: sequence by host.id with maxspan=3m [process where process.name=="wscript.exe"] [process where process.name=="cmd.exe"] [network where true]. KQL cannot express ordering between events; it just filters a pool of events. ES|QL aggregates but doesn't sequence.
QUESTION 04
A threat hunter wants to search for events where the source.ip field contains ANY value (i.e., the field exists and is populated). What is the correct KQL syntax?
Correct: source.ip: * — In KQL, using a wildcard * as the value matches any document where that field has any value (i.e., the field exists and is not null). This is the standard KQL idiom for "field exists." Option D (_exists_) is Lucene syntax, not KQL. Options A and B are not valid KQL syntax.
QUESTION 05
During a threat hunt, an analyst finds suspicious events and wants to document findings, pin key events, write analyst notes, and share with the team for a collaborative investigation. Which feature should they use?
Correct: Create a Timeline and attach to a Case. Timelines are the purpose-built investigation documentation tool — they support pinned events, per-event analyst notes, query history, and team sharing. Attaching to a Case (Topic 6) creates a full incident record. Creating a detection rule is also a good outcome but doesn't document the current hunt finding. CSV export and saved searches lose context and collaboration features.
QUESTION 06
What is the correct ES|QL syntax to count the number of unique destination IPs per source host from network logs?
FROM logs-network*
| WHERE event.category == "network"
| _____ unique_dests = ________(destination.ip) BY host.name
| SORT unique_dests DESC
Correct: STATS unique_dests = COUNT_DISTINCT(destination.ip) BY host.name — ES|QL uses STATS for aggregations (not GROUP BY — that's SQL). COUNT_DISTINCT() counts unique values. The correct full line is: | STATS unique_dests = COUNT_DISTINCT(destination.ip) BY host.name. GROUP BY, AGGREGATE, and SELECT are SQL keywords — ES|QL has its own command set.
QUESTION 07
An analyst uses Osquery to interrogate a live endpoint during a hunt. What is a key difference between Osquery results and Elastic Security event data in Discover?
Correct: Osquery captures live volatile state; Discover searches historical events. Osquery queries the endpoint in real-time — it sees currently running processes, active network connections, and current file state. This is volatile data that may not have been logged. Discover searches events that have already been ingested into Elasticsearch. Both are complementary: Discover for historical investigation, Osquery for current endpoint state verification. Osquery uses SQL syntax and works on Windows, Linux, and macOS.
