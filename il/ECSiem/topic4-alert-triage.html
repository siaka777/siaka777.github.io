04.01

Alert Triage Overview

Alert triage is the systematic process of reviewing, prioritizing, and classifying security alerts. In Elastic Security, the Alerts page (Security → Alerts) is the primary workspace for SOC analysts. Effective triage separates signal from noise and ensures high-severity threats receive immediate attention.

The triage workflow follows a structured sequence — from initial receipt through investigation to final disposition. Understanding this flow is fundamental to the ECSA exam.

Where alerts live: Alerts are stored in .alerts-security.alerts-default (or space-specific variant). They are NOT the same as the original event data. The alert document contains a copy of key event fields plus rule metadata, status, and analyst notes.

04.02

Alert Statuses & Workflow States

Every alert in Elastic Security has a status that tracks where it is in the triage lifecycle. Understanding the four statuses — and when to use each — is a frequent exam topic.

StatusMeaningWhen to UseWho Acts
OPEN New, unreviewed alert Default state when alert fires. Needs triage. System (auto)
ACKNOWLEDGED Seen, under review Analyst has seen it but investigation is not complete. Analyst
IN PROGRESS Actively being worked Investigation is active. Analyst is building a timeline or case. Analyst
CLOSED Resolved with verdict Fully triaged. Must select a closing reason (see below). Analyst / Lead

When closing an alert, analysts must select a closing reason that documents the verdict:

🚨
True Positive
Confirmed malicious activity. Escalate or contain.
✅
False Positive
Alert fired but activity is legitimate. Rule tuning may be needed.
⚠️
Benign True Positive
Real behavior the rule caught, but not malicious in this context.
❓
Indeterminate
Insufficient data to make a verdict. Needs more investigation.
← Click a verdict card above to see exam notes and recommended next actions

Exam trap: "Benign True Positive" (BTP) is different from "False Positive." A BTP is when the rule correctly detected the behavior, but that behavior is acceptable in this environment (e.g., a sysadmin running mimikatz during an authorized pen test). A FP is when the rule fired on activity that doesn't match the intended detection at all.

Triage prioritization rule: Always triage alerts by severity + recency + asset criticality. A Critical alert on a Domain Controller (WIN-DC-01) takes absolute priority over Medium alerts on standard workstations. Risk score helps quantify this — sort the alert queue by risk score descending as a starting point.

04.04

Investigation Tools & Workflows

Once an alert is selected for investigation, Elastic Security provides several interconnected tools to build context, correlate events, and document findings.

🔍
Step 1
Expand the Alert — Analyzer & Details
Click any alert to open the alert details flyout. Review the triggering event fields, the rule that fired, and the MITRE ATT&CK context. The Analyzer tab shows the process tree graphically — parent/child process relationships rendered as a visual graph. Essential for understanding lateral movement and process injection.
📋
Step 2
Review Host & User Context
The Host Risk Score and User Risk Score panels (Security → Entity Analytics) show historical risk levels. A host that has been involved in multiple alerts recently gets a higher risk score. This helps prioritize which alerts from a busy host truly need attention versus routine noise.
⏱️
Step 3
Build a Timeline
Click "Investigate in Timeline" from an alert to automatically create a timeline pre-populated with the alert's event. Timelines are saved investigation notebooks — analysts add more events, write notes using the ECS-aware query bar, and pin important events. Timelines can be attached to Cases for documentation.
🔗
Step 4
Correlate Related Events
From the alert flyout, use "View related events" to search Discover for events from the same host within a time window. Use KQL/EQL in Discover to pivot on IOCs — search for other hosts communicating with the same IP, other users running the same process, or other events from the same user within the incident window.
📁
Step 5
Create / Attach to a Case
Once the alert is confirmed True Positive, click "Add to existing case" or "Create new case". The alert and any attached timelines become part of the case. Cases support collaborators, tags, connectors (JIRA, ServiceNow), and status tracking. Covered fully in Topic 6.

04.05

Entity Analytics & Risk Scoring

Elastic Security's Entity Analytics section (Security → Entity Analytics) provides aggregated risk views for hosts and users. It uses alert history and ML signals to build risk scores that persist over time, helping analysts identify high-risk entities even when individual alerts are low-severity.

Host Risk Score
Aggregated score based on:
• Alert severity history
• Number of unique rules fired
• ML anomaly contributions

Score ranges: Low · Medium ·
High · Critical

Updated on each new alert event.
User Risk Score
Same model as host, but scoped
to user.name across all systems.

A user who triggered alerts on
5 different hosts in 24 hours
gets a much higher risk score
than their individual alerts suggest.

Used for insider threat detection.

Exam scenario: An analyst sees 20 low-severity alerts from various systems. Individually, none are alarming. But Entity Analytics shows one user has a Critical risk score because they triggered alerts on 8 different hosts. This is a strong lateral movement indicator — the Entity Analytics view surfaced the threat that individual alert triage would have missed.

Alert Filtering, Sorting & Bulk Actions

The Alerts page provides powerful filtering to help analysts manage high-volume alert queues. Knowing which filters to apply in which scenario is a key exam skill.

Key exam concept: The Alerts page searches .alerts-security.alerts-*, which is separate from raw event indices. Alert fields use the kibana.alert.* namespace for Kibana-added metadata, while the original event fields (like process.name) are copied directly from the source event into the alert document.

Topic 4 · Practice Questions
QUESTION 01
An analyst receives an alert for a known IT administration tool (psexec.exe) running on a server where it is regularly used by the IT team. The rule correctly detected psexec, but the activity is authorized. What is the correct closing reason?
Correct: Benign True Positive (BTP). BTP means the rule correctly detected the behavior it was designed to detect, but in this specific context the behavior is legitimate. The rule fired correctly on psexec.exe — it's working as intended. A False Positive would mean the rule fired on something that isn't actually psexec-like behavior at all. After closing as BTP, the analyst should add an exception to reduce future noise for this specific authorized scenario.
QUESTION 02
An analyst wants to visually see the parent-child process relationships for a suspicious process that triggered a Critical alert. Which Elastic Security feature provides this graphical view?
Correct: The Analyzer tab. When you open an alert's detail flyout, the Analyzer tab renders a graphical process tree showing the entire process lineage — from the root process down through all child processes. Nodes are color-coded by suspiciousness and clicking any node shows its full ECS event data. This is invaluable for understanding attack chains like macro → cmd → powershell → network.
QUESTION 03
What is the correct sequence of alert workflow states in Elastic Security from initial detection to resolution?
Correct: Open → Acknowledged → In Progress → Closed. These are the four official Elastic Security alert statuses. Open (new/unreviewed) → Acknowledged (seen, review pending) → In Progress (actively investigating) → Closed (resolved with a closing reason: True Positive, False Positive, Benign True Positive, or Indeterminate). Memorize these exact terms.
QUESTION 04
An analyst confirms a Critical alert on a Windows host is a True Positive ransomware infection. The endpoint has Elastic Defend installed. What is the immediate containment action available directly from the Elastic Security UI?
Correct: Isolate the host via Elastic Defend. When Elastic Defend is installed on an endpoint, analysts can trigger host isolation directly from the alert flyout in Kibana. Isolation cuts all network connectivity except for the Elastic Agent management channel, containing the threat while preserving forensic state. This requires the analyst to have the appropriate Elastic Security role with endpoint response capabilities.
QUESTION 05
An analyst is pivoting during an investigation and wants to find all other hosts that made network connections to the same destination IP as the alert. Which approach is correct in Elastic Security?
Correct: Pivot to Discover or Timeline and query raw event data. Alerts only contain the events that triggered a rule. To find ALL hosts that communicated with a specific IP — including hosts that haven't triggered any rule — you need to search the underlying event indices (logs-endpoint*, logs-network*, etc.) using destination.ip: "x.x.x.x" in Discover or a Timeline. The alert queue only shows alerted events, not the full picture.
QUESTION 06
An analyst needs to document an investigation by collecting multiple related alerts, query results, and analyst notes into a single trackable record. Which feature should they use?
Correct: Create a Case. Cases (Security → Cases) are the dedicated incident management feature in Elastic Security. They support attaching multiple alerts, timelines, comments, assignees, tags, and external ticketing system connectors (Jira, ServiceNow). This is the official workflow for incident documentation — not tags or CSV exports. Full coverage in Topic 6.
QUESTION 07
An analyst sees 15 low-severity alerts from different hosts over the past hour. Individually, none are alarming. What Elastic Security feature would help identify if these alerts indicate a coordinated attack campaign by a single user?
Correct: Entity Analytics — User Risk Score. Entity Analytics aggregates all alert activity across every host for each user and calculates a cumulative risk score. A user who triggered 15 low-severity alerts across 8 different hosts in one hour would have a very high User Risk Score, flagging them as a high-priority investigation target. This aggregated view is what makes Entity Analytics powerful — it sees the campaign pattern that individual alert triage would miss.
