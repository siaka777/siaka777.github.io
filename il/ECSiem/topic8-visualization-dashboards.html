08.01

Visualization in Elastic Security

Kibana's visualization layer transforms raw security event data into actionable dashboards, charts, and reports. For the ECSA exam, you need to know which visualization tool to use, when to use Discover vs Lens vs Canvas vs Maps, and how to build effective security dashboards using the right panel types.

Elastic Security ships with pre-built dashboards for key integrations (Endpoint, Network, AWS, etc.), but analysts must understand how to create and customize dashboards for their environment, and how to filter, drill-down, and link visualizations to investigations.

Core Visualization Tools
Lens — drag-drop viz builder (primary)
Discover — raw event exploration + charts
Maps — geospatial IP/threat visualization
Canvas — pixel-perfect presentation reports
TSVB — time-series metric panels
Vega — custom D3-based visuals (advanced)
Pre-Built Security Dashboards
Found in Security → Dashboards:
• Overview — alert summary by severity
• Network — traffic flows, DNS, geo
• Host — process, file, registry events
• User — authentication, activity
• Cloud — cloud provider activity
• Entity Analytics — risk scores

Exam tip — Discover vs Dashboards: Discover is for ad-hoc, interactive investigation of raw events. Dashboards built with Lens are for persistent monitoring and team-wide situational awareness. Security analysts use both: Discover when hunting/investigating a specific alert, pre-built dashboards for ongoing SOC monitoring.

Bar / Column Chart
WHEN TO USE: Compare discrete categories (rule names, host names, process names) or time-bucketed counts. SECURITY USE CASES: • Alert count by rule name — identify your noisiest rules • Events by host.name — spot compromised or busy hosts • Authentication failures by user.name — brute force visibility • Process execution counts by process.name — baseline deviation LENS CONFIGURATION (Vertical Bar): X-axis → Date Histogram (@timestamp) or Terms (host.name) Y-axis → Count or Sum(numeric_field) Break down by → severity (for stacked bar by severity level) EXAM TIP: For "which hosts triggered the most alerts in the past 24 hours?" → Vertical bar chart, X = host.name (Terms, Top 10), Y = Count
Line / Area Chart
WHEN TO USE: Show trends over time. Spot spikes, anomalies, drops in detection coverage. SECURITY USE CASES: • Alert volume trend — is our attack surface growing? • Detection rate over time — are rules catching more over time? • Failed login attempts over 7 days — identify brute force windows • Network bytes per hour — detect exfiltration spikes LENS CONFIGURATION: X-axis → Date Histogram (@timestamp), interval = auto/1h/1d Y-axis → Count, Avg(risk_score), Sum(network.bytes) Line series → one per severity level (for multi-line trend) EXAM TIP: Date Histogram is always the X-axis for time-series line charts. "Show alert trend over the last 30 days" = line chart + date histogram.
Donut / Pie Chart
WHEN TO USE: Show proportional composition when you have a small number of categories (≤8). Avoid for many categories — use a bar chart or table instead. SECURITY USE CASES: • Alert severity distribution — what % are Critical vs High vs Medium? • Event category breakdown — process vs network vs authentication • MITRE tactic distribution — which tactics are most active? • Alert status breakdown — Open vs Acknowledged vs Closed LENS CONFIGURATION: Slice by → Terms (kibana.alert.severity, event.category) Metric → Count Max slices → 5–8 recommended EXAM TIP: Donut/Pie is for PROPORTIONS of a finite set. If someone asks "what percentage of alerts are Critical?" → Donut chart.
Data Table
WHEN TO USE: Show ranked lists with multiple metrics per entity. Tables are essential when you need multiple columns of data per row. SECURITY USE CASES: • Top 10 hosts by alert count + risk score + last seen • Top processes by execution count + unique hosts + unique users • Failed auth attempts per user + source IP + target host • Rule performance: name + fires/day + FP rate + severity LENS CONFIGURATION: Row → Terms aggregation (host.name, user.name, process.name) Cols → Count, Sum, Avg, Top Values (additional metrics) Sort → Descending by count EXAM TIP: "Show the top 10 hosts ranked by most alerts with their risk scores" → Data Table, not a bar chart. Tables = multiple metrics per row.
Metric / KPI Panel
WHEN TO USE: Display a single prominent number as a KPI — overview dashboards and executive reports. SECURITY USE CASES: • Total open alerts (count) • Critical alert count today • Mean Time to Detect (MTTD) in minutes • Mean Time to Respond (MTTR) in hours • Active hosts with endpoint coverage • Rules currently enabled LENS CONFIGURATION: Primary metric → Count (with filter: status = "open") Secondary metric → optional trend/comparison value EXAM TIP: Metric panels are for at-a-glance KPIs. "Display the total count of open critical alerts as a single number" → Metric panel. Can use color thresholds: green < 10, amber 10–50, red > 50.
Heat Map
WHEN TO USE: Visualize density across two dimensions — time-of-day patterns, host × rule density. SECURITY USE CASES: • Alerts by hour-of-day × day-of-week — spot attack time patterns • Authentication failures by user × host — lateral movement patterns • Process launches by host × process.name — which hosts run what? • Alert density by MITRE tactic × time period LENS CONFIGURATION: X-axis → Hour of day (scripted field or date format: HH) Y-axis → Day of week Cell value → Count Color → gradient (low = blue, high = red) EXAM TIP: Heat maps reveal TEMPORAL PATTERNS that line charts obscure. "Show which hours of the day have the most authentication failures" → Heat map (hour × day).
Kibana Maps
WHEN TO USE: Geospatial analysis of IP addresses, threat origins, network flows. SECURITY USE CASES: • Source IP geolocation of authentication failures • Destination countries for outbound C2 connections • Threat intelligence indicator geographic distribution • VPN connection source visualization • Attack origin heatmap (clustered point layer) CONFIGURATION: Layer type → Documents (for individual events) or Grid aggregation (for density) Geo field → source.geo.location or destination.geo.location Filter → event.category: "authentication" AND event.outcome: "failure" EXAM TIP: Kibana Maps uses ECS geo.location fields (geo_point type). Requires source.geo.* or destination.geo.* to be populated (via geoip processor). "Show a map of where failed logins are coming from" → Kibana Maps.
ES|QL Table / Visualization Panel
WHEN TO USE: When you need aggregated analytics or computed metrics that Lens cannot express (e.g., COUNT_DISTINCT, computed ratios, complex multi-step aggregations). SECURITY USE CASES: • Top 10 rarest processes (COUNT_DISTINCT hosts per executable) • Users with most unique source IPs (potential account sharing) • Alert rate per rule per hour (computed in ES|QL, displayed as table) • DNS query volume ratio (total/unique — beacon detection) CONFIGURATION: In Lens: Switch to "ES|QL" query mode Write the full ES|QL query (FROM … | WHERE … | STATS … | SORT …) Output renders as table or chart depending on query structure EXAM TIP: ES|QL panels are displayed within Dashboards just like Lens panels. They are ideal when STATS aggregations like COUNT_DISTINCT or EVAL-computed fields are needed — operations impossible in standard Lens drag-drop mode.

Lens — Interactive Preview

Kibana Lens is the primary visualization builder. It uses a drag-and-drop interface to build charts from aggregated field data. Understanding Lens concepts — data views, aggregations, buckets, and chart switching — is essential exam knowledge.

// Lens Builder Simulator

Metric (Y)

Key Lens concepts: A Data View (formerly "Index Pattern") maps to one or more index patterns (e.g., .alerts-security.alerts-*). The Date Histogram bucket splits data by time and is used for time-series charts. Top Values bucket groups by a field's top N terms. All aggregations run against Elasticsearch — Lens is a GUI over ES aggregation queries.

// Behind every Lens panel is an Elasticsearch aggregation request // This is what Lens generates for "Alert count by severity (Top 5)": { "aggs": { "2": { "terms": { "field": "kibana.alert.severity", "size": 5, "order": { "_count": "desc" } }, "aggs": { "1": { "count": {} } } } }, "query": { "bool": { "filter": [{ "range": { "@timestamp": { "gte": "now-24h" } } }] } } }
08.04

Dashboard Design Scenarios

The ECSA exam presents dashboard design scenarios: given a requirement, which visualization type and configuration is correct? Click each scenario to see the recommended approach.

01
SOC Manager · Situational Awareness
"I need to see total open alerts, how many are Critical, and the trend over the last 7 days — all at a glance."
Executive overview dashboard design
RECOMMENDED APPROACH: Panel 1 (top-left): Metric panel → COUNT where status="open" Panel 2 (top-centre): Metric panel → COUNT where severity="critical" Panel 3 (top-right): Metric panel → COUNT where severity="high" Panel 4 (bottom): Line chart → Date Histogram (7 days), Y = Count, Break down by: severity COLOR THRESHOLDS on Metric panels: Total open: green <50, amber 50-200, red >200 Critical: green = 0, amber 1-5, red >5 EXAM TIP: Multiple Metric panels + one time-series line chart is the standard SOC overview layout. Metric panels give instant KPI status; line chart shows trends.
02
Threat Intelligence · Geo Analysis
"Show me where failed authentication attempts are geographically originating from in the last 24 hours."
Geographic threat visualization
RECOMMENDED APPROACH: Tool: Kibana Maps (not Lens) Layer: Documents layer OR Grid aggregation layer Index: logs-* (auth events) Filter: event.category: "authentication" AND event.outcome: "failure" Geo field: source.geo.location (geo_point type) PREREQUISITES: → source.geo.* fields must be populated (via geoip ingest processor) → The index mapping must have source.geo.location as geo_point type LAYER TYPES: Documents → individual event pins (good for <10k events) Grid aggregation → density heatmap by region (good for high volume) Point-to-point → show connections between source and destination IPs EXAM TIP: Any question about geographic visualization of IPs = Kibana Maps. Lens cannot render maps. The geoip processor must be in the ingest pipeline.
03
Detection Engineering · Rule Performance
"Which detection rules are firing the most alerts, and what's the average risk score per rule?"
Rule performance analysis dashboard
RECOMMENDED APPROACH: Tool: Lens → Data Table Index: .alerts-security.alerts-* Rows: Terms aggregation on kibana.alert.rule.name (Top 20) Column 1: Count (total alerts) Column 2: Average of kibana.alert.risk_score Column 3: Max of kibana.alert.severity (for worst severity seen) Sort: Descending by Count ALTERNATIVE (ES|QL for more control): FROM .alerts-security.alerts-* | WHERE @timestamp > NOW() - 7 days | STATS alert_count = COUNT(*), avg_risk = AVG(kibana.alert.risk_score) BY kibana.alert.rule.name | SORT alert_count DESC | LIMIT 20 EXAM TIP: Data Tables with multiple metrics per row are best for rule performance. "Top N rules ranked by X with metrics Y and Z" = Data Table, not bar chart.
04
Behavioral Analysis · Temporal Patterns
"Show whether attackers are working specific hours — when are most alerts firing during the week?"
Temporal attack pattern visualization
RECOMMENDED APPROACH: Tool: Lens → Heat Map Index: .alerts-security.alerts-* X-axis: Hour of day (computed: hour_of_day from @timestamp) Y-axis: Day of week (computed: day_of_week from @timestamp) Value: Count Color: Gradient low=blue, high=red HOW TO CREATE HOUR/DAY FIELDS: Use a scripted field in the data view: hour_of_day → doc['@timestamp'].value.hourOfDay day_of_week → doc['@timestamp'].value.dayOfWeekEnum.toString() OR use ES|QL with DATE_EXTRACT(): FROM .alerts-security.alerts-* | EVAL hour = DATE_EXTRACT("HOUR_OF_DAY", @timestamp) | EVAL dow = DATE_EXTRACT("DAY_OF_WEEK", @timestamp) | STATS count = COUNT(*) BY hour, dow | SORT count DESC EXAM TIP: Heat map = 2D density analysis. The hour-of-day × day-of-week heat map is the classic "working hours vs off-hours attacker" visualization.
05
Dashboard Filters · Drill-Down
"When I click a bar in the host chart, I want all other panels to filter to just that host automatically."
Dashboard interactivity and cross-filtering
RECOMMENDED APPROACH: Feature: Dashboard controls + Cross-panel filtering (built-in to Kibana Dashboards) How it works: 1. All panels on the same dashboard share the same time range and global filters 2. Clicking a value in most Lens panels automatically creates a KQL filter that applies to ALL panels on the dashboard 3. The filter appears in the KQL bar at the top: host.name: "WIN-DC-01" ADDITIONAL CONTROLS: • Option Lists → dropdown selector for host.name, severity, etc. • Range Sliders → numeric range filter (e.g., risk_score slider) • Time Slider → scrub through time ranges → All controls filter ALL panels on the dashboard simultaneously DRILL-DOWN: Right-click a chart element → "Drill down" or configure URL-based drilldowns to navigate from dashboard → Discover with filters pre-applied EXAM TIP: Cross-panel filtering is AUTOMATIC in Kibana dashboards — clicking a value in one panel filters all others. No special configuration needed. Controls are optional additions for pre-built filter selectors.

Dashboard best practices for the exam: Pre-built Elastic Security dashboards (Security → Dashboards) cannot be edited — they must be cloned first before customization. To save a custom dashboard, you must be in edit mode. Dashboards are Space-scoped saved objects — they do not automatically appear in other Spaces.

Final Visualization Quiz
QUESTION 01
A SOC manager wants a dashboard that shows the total count of open critical alerts as a large, prominent number that changes colour to red when it exceeds 10. Which visualization type should be used?
Correct: Metric panel with colour thresholds. Metric panels are purpose-built for single prominent KPI values. They support colour thresholds: configure green when <10, red when ≥10. The filter would be kibana.alert.severity: "critical" AND kibana.alert.workflow_status: "open". A line chart shows trends (not a current count). A table shows rows of data. A donut shows proportions, not a single highlighted number.
QUESTION 02
An analyst clicks on a bar labelled "WIN-DC-01" in a host alert-count bar chart on a dashboard. What happens to the other panels on the same dashboard?
Correct: All other panels filter to WIN-DC-01. Kibana Dashboard cross-panel filtering is automatic — clicking a value in any Lens panel creates a KQL filter that applies to the entire dashboard. All panels sharing the same data view will update to show only events from WIN-DC-01. The filter appears in the KQL bar at the top of the dashboard and can be removed or pinned.
QUESTION 03
Which visualization is BEST suited to answer: "At what hours of the day and days of the week do authentication failures peak in our environment?"
Correct: Heat map with hour × day. A heat map visualizes density across two dimensions simultaneously. With hour-of-day (0–23) on X and day-of-week on Y, the colour intensity of each cell shows peak activity times — revealing patterns like "most attacks happen Monday-Friday, 02:00-05:00 UTC." A line chart shows one dimension over time. A bar chart shows one dimension. Only a heat map combines two categorical dimensions with density colour coding.
QUESTION 04
An analyst wants to display where failed login source IPs are geographically located on a world map. Which Kibana feature must they use, and what is required for it to work?
Correct: Kibana Maps, requiring geo_point fields from the geoip processor. Kibana Maps is the dedicated geospatial visualization tool. It requires the events to have source.geo.location (type: geo_point) populated. These fields are typically added by the geoip ingest pipeline processor which translates IP addresses to coordinates. Lens does not have a map chart type. Discover has no map view.
QUESTION 05
A security analyst wants to modify one of the pre-built Elastic Security dashboards (under Security → Dashboards) to add a custom panel. What must they do first?
Correct: Clone the dashboard first. Pre-built Elastic Security dashboards (Security → Dashboards) are read-only — they cannot be directly edited to prevent overwriting official content during upgrades. To customize them, the analyst must use the "Clone" option, which creates an editable copy in Analytics → Dashboards within the same Space. Roles don't unlock read-only status. Space-switching does not change the read-only nature of pre-built content.
QUESTION 06
An analyst needs a dashboard panel that shows: the top 20 process executables, how many unique hosts ran each one, and how many total executions occurred. Which visualization type is most appropriate?
Correct: Data table. When you need multiple metrics per entity (unique hosts AND total count per process), a Data Table is the right choice. Configure: Rows = Terms(process.executable, Top 20), Column 1 = Count, Column 2 = Count Distinct(host.name), sorted by Count descending. A bar chart can only show one metric per bar. A donut shows proportions of a single metric. A line chart shows time trends.
QUESTION 07
What is the correct location in Kibana to find dashboards that came pre-installed with the Elastic Security integration (like the "Overview" and "Network" dashboards)?
Correct: Security → Dashboards. The pre-built security dashboards (Overview, Network, Host, User, Cloud, Entity Analytics) are accessed through the dedicated Security → Dashboards section in Elastic Security. While they are technically also visible in Analytics → Dashboards (as tagged read-only objects), the primary designed access point and exam-expected answer is Security → Dashboards. This section also filters to show only security-relevant dashboards, not all Kibana dashboards.
