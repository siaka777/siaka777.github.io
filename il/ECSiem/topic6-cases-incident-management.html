06.01

What Are Cases?

Cases in Elastic Security are the incident management layer — a structured workspace for documenting security incidents, tracking investigation progress, collaborating with teammates, and integrating with external ticketing systems. Cases live at Security → Cases.

Think of a Case as the official incident record that ties together: the triggering alerts, investigation timelines, analyst notes, evidence, assigned owners, and final resolution — all in one place. Cases are scoped to a Kibana Space, so each team maintains their own incident records.

Cases vs. Alerts vs. Timelines: These three are related but distinct:
  • Alert — a single detection event generated by a rule
  • Timeline — an investigation notebook where you query and organize events
  • Case — the formal incident record that contains alerts, timelines, and comments

06.02

Case Statuses & Lifecycle

Cases progress through four statuses during their lifecycle. Unlike alert statuses, case statuses are manually managed by analysts and reflect the overall investigation state — not individual alert disposition.

Open
New case created.
Investigation not started.
In Progress
Actively being investigated by assigned analyst.
Closed
Investigation complete. Incident resolved.
StatusDescriptionWho UpdatesTypical Duration
Open Default state on creation. Case needs attention and assignment. System / Creator Minutes–hours (should be picked up quickly)
In Progress Active investigation. Analyst is gathering evidence, building timelines. Assigned Analyst Hours–days (proportional to incident complexity)
Closed All investigation tasks complete. Incident contained and documented. Analyst / Lead Permanent (can be re-opened)

Case severity vs. Alert severity: Case severity is independently set from the alert severity. A case can be created from a High-severity alert but classified as Critical at the case level if the investigation reveals broader impact. The four case severity levels mirror alerts: Critical, High, Medium, Low.

06.03

Creating & Managing Cases

Cases can be created from three starting points. Understanding each entry point is important for exam scenarios about workflow.

// Three ways to create a Case in Elastic Security 1. FROM AN ALERT Security → Alerts → [select alert] → Actions → "Add to existing case" or "Create new case" // Most common path. Alert is automatically attached. 2. FROM A TIMELINE Security → Timelines → [open timeline] → Attach to Case // Useful when investigation started in Timeline before case created. 3. DIRECTLY Security → Cases → Create case // Used for proactive tracking or when starting from external report. // Attach alerts and timelines afterwards.

When creating a case, the following fields are configured:

Required Fields
Title — descriptive case name
Severity — Critical/High/Medium/Low
Owner — Kibana Space that owns the case

Optional but recommended:
Description, Assignees, Tags,
External Connector mapping
Post-Creation Actions
Add alerts — from Alerts page or flyout
Add timelines — from Timelines page
Add comments — analyst notes with markdown
Upload files — evidence attachments
Sync to connector — push to Jira/SNOW
Assign/reassign — change analyst owner

Exam critical: You can add the same alert to multiple cases. You can add multiple alerts to one case. Alerts are not consumed or removed when added to a case — the original alert remains in the Alerts page. The case stores a reference to the alert, not a copy.

06.04

Timelines — Investigation Notebooks

A Timeline is Elastic Security's investigation notebook. It provides a chronological workspace where analysts query raw event data, pin key events, write notes, and document the sequence of events in an incident. Timelines are attached to cases to become part of the formal incident record.

06.05

External Connectors & Integrations

Cases can be connected to external ticketing and notification systems via Connectors. When a case has an active connector, changes in Elastic Security can be pushed (synced) to the external system automatically, and external ticket IDs are displayed in the case record.

Connectors are configured at Stack Management → Connectors and then selected when creating or editing a case.

Topic 6 — Knowledge Assessment
QUESTION 01
An analyst confirms a True Positive alert and wants to begin formal incident tracking. They need to attach two additional related alerts and a timeline to the same record. What should they create?
Correct: Create a Case and attach the alerts and timelines. Cases are the formal incident management construct in Elastic Security. They aggregate multiple alerts, timelines, comments, and file evidence into a single trackable record. Saved searches and exception lists serve different purposes. This is the canonical workflow for TP incident tracking.
QUESTION 02
What is the key difference between attaching a Timeline to a Case versus adding an Alert to a Case?
Correct: Both store references — originals remain. When you attach an alert or timeline to a case, Elastic Security stores a reference, not a copy. The original alert remains in the Alerts page (and can be attached to multiple cases). The original timeline remains in Security → Timelines. Nothing is deleted or moved. This means one alert can appear in multiple cases if it's relevant to several incidents.
QUESTION 03
A SOC manager wants to automatically create a Jira ticket every time a Critical case is created in Elastic Security. Which feature enables this integration?
Correct: Jira Connector in Stack Management. Connectors (Stack Management → Connectors) are the built-in integration mechanism for external systems. A Jira Connector stores the API credentials and project settings. Once configured and assigned to a case, analysts can push the case to Jira with one click. Detection rule actions can also use connectors for alert-level notifications, but case-level integration uses the Cases connector setting.
QUESTION 04
An analyst updates a case status to "Closed" in Elastic Security after resolution. The case had a Jira connector configured. What happens to the linked Jira ticket?
Correct: Nothing happens automatically. Connectors in Elastic Security Cases are manually triggered — you click "Push as new incident" or "Update incident" to sync. Status changes in the Elastic Security case do NOT automatically propagate to Jira (or vice versa). This is a common exam trap. Bidirectional real-time sync would require custom webhook automation outside of Elastic's built-in connector functionality.
QUESTION 05
An analyst working in the "Cloud-SOC" Kibana Space tries to find a case that was created in the "Windows-SOC" space. What will they see?
Correct: Cases are space-scoped and not visible in other spaces. All Kibana saved objects — including cases, timelines, detection rules, and dashboards — are scoped to a specific Kibana Space. An analyst in "Cloud-SOC" space cannot see cases created in "Windows-SOC" space. They would need to switch spaces (if they have access) to view those cases. This space isolation is intentional for multi-team SOC environments.
QUESTION 06
Which case feature allows a security analyst to attach structured IOC data (such as a C2 IP address or malware hash) directly to a case record for future filtering and correlation?
Correct: Case Observables. Observables are structured IOC fields in a case that accept typed values like IP addresses, domain names, file hashes, URLs, and email addresses. They're distinct from tags (which are free-form labels) and comments (which are unstructured text). Observables enable searching across all cases that share a specific IOC — e.g., "show me all cases where 185.234.x.x was observed" — which is invaluable for campaign correlation.
QUESTION 07
What is the primary function of a Timeline in Elastic Security, and how does it differ from the Alerts page?
Correct: Timelines query raw event data; Alerts page shows rule-triggered alert documents. The Alerts page searches the .alerts-security.alerts-* index — only events that triggered a detection rule. Timelines search the underlying raw event indices (logs-*, filebeat-*, etc.) using KQL or EQL. This means analysts can build complete incident narratives in Timelines using raw events — including events that never triggered any rule — providing full investigation coverage beyond just what was alerted.
