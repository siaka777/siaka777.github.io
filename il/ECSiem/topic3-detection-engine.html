03.01

The Detection Engine

The Elastic Security Detection Engine runs continuously against indexed security data, evaluating detection rules on a scheduled basis. When a rule's conditions are met, it generates an alert in the .alerts-security.alerts-* data stream.

Detection rules are the operational heart of a SOC — they encode threat knowledge into repeatable, automated detection logic. The exam focuses heavily on understanding the different rule types, when to use each, and how to configure them correctly.

How Rules Execute
Rules run on a schedule (e.g., every 5 min).
Each execution queries a look-back window.
Matches create alerts with severity, risk score,
and MITRE ATT&CK mappings.
Rules can be enabled / disabled / snoozed.
Alert Storage
Alerts go to:
.alerts-security.alerts-default

They are separate from raw event indices.
Analysts triage alerts in Security → Alerts.
Alerts persist until closed or deleted.

Key concept: Rules do NOT modify original event data. They generate new alert documents that reference the triggering event. The original log remains unchanged in its source index.

03.03

Anatomy of a Detection Rule

Every rule shares a common set of configuration fields. Understanding each field is critical — many exam questions describe a scenario and ask you to identify which setting needs to be changed.

// Detection Rule — Core Fields Reference name: "Suspicious PowerShell Encoded Command" description: "Detects use of -EncodedCommand flag in PowerShell" rule_type: "query" // query | eql | threshold | machine_learning | threat_match | new_terms language: "kuery" // kuery | lucene | eql | esql query: "process.name: powershell.exe AND process.args: (*EncodedCommand* OR *-enc*)" // Scheduling interval: 5m // How often the rule runs from: "now-6m" // Look-back window start (should overlap interval) to: "now" // Risk & Severity severity: "high" // critical | high | medium | low risk_score: 73 // 0–100, used to sort/prioritize alerts // MITRE ATT&CK mapping threat: [{ tactic: { id: "TA0002", name: "Execution" }, technique: [{ id: "T1059.001", name: "PowerShell" }] }] // Actions (optional — notify on alert) actions: [{ action_type_id: ".slack", params: { message: "Alert: {{rule.name}}" } }]
Interval vs Look-Back
Interval = how often rule runs (e.g., 5m)
Look-back (from) = window of data searched

Best practice: look-back should be slightly longer than interval to prevent gaps.
E.g., interval=5m → from=now-6m
Risk Score
Numeric value 0–100.
Used to sort and filter alerts.

Severity is qualitative (critical/high/medium/low).
Risk score is quantitative — combine both for prioritization.

Alert severity levels and typical risk scores:

CRITICAL  91–100
HIGH  73–90
MEDIUM  47–72
LOW  21–46
INFORMATIONAL  0–20
03.04

Event Query Language (EQL)

EQL is a purpose-built language for detecting sequences, patterns, and relationships across multiple events. It's used by Event Correlation rules and is the most powerful rule type for detecting multi-step attack chains.

// EQL — Three query types you must know // 1. SEQUENCE — ordered chain of events (most important for SIEM) sequence by host.id with maxspan=1m [process where process.name == "cmd.exe"] [process where process.name == "net.exe" and process.args : "user"] [network where destination.port == 4444] // → Detects: cmd spawns net user, then connects to C2 port — within 1 minute on same host // 2. EVENT — single event match (like KQL but typed) process where process.name == "mimikatz.exe" and event.type == "start" // 3. ANY — matches across all event categories any where network.direction == "egress" and destination.port in (22, 443, 8080) and not source.ip : "10.0.0.0/8"
EQL Event Categories
process — process start/stop events
network — connection events
file — file create/modify/delete
registry — Windows registry changes
dns — DNS queries
any — any event type
EQL Key Syntax
by field → group sequence by field
with maxspan=Xm → time window
until → stop sequence on event
: "glob*" → wildcard match
in (a, b, c) → multi-value match
not → negate a condition

Exam trap: EQL uses == for equality (not : like KQL). It also requires an event category prefix (process where, network where). Mixing KQL and EQL syntax is a common exam distractor.

03.06

Rule Tuning & Exceptions

Out-of-the-box rules often generate false positives — legitimate activity that matches the detection logic. Tuning rules to reduce noise without losing true positives is a core analyst skill.

// Exception Types in Elastic Security Rule Exception → Applies to ONE specific rule only Shared Exception List → Applies to MULTIPLE rules (shared across space) Endpoint Exception → Passed down to Elastic Defend agent (prevents event generation) // Example: Exception entry to suppress FP from IT admin tool { field: "process.name", operator: "is", value: "psexec.exe", field: "user.name", operator: "is", value: "svc-deployment", field: "host.name", operator: "is one of", value: ["deploy-srv-01", "deploy-srv-02"] } // → This suppresses alerts ONLY when all three conditions are true simultaneously
Tuning Strategies
Add exceptions — whitelist known-good
Raise risk threshold — threshold rules
Refine query — add exclusion clauses
Adjust look-back — reduce noise window
Alert suppression — deduplicate alerts
Rule schedule — run less frequently
Alert Suppression
Suppresses duplicate alerts within
a time window when the same
field value triggers repeatedly.

E.g., suppress by host.name for 1h
→ 1 alert per host per hour max,
even if rule fires 100 times.
// Interactive Rule Preview Builder

Critical exam concept: Endpoint exceptions are the most powerful — they prevent the event from being generated on the agent entirely. Use them carefully. Rule exceptions suppress the alert but the event is still indexed. Shared exception lists help maintain consistency across many rules.

Topic 3 · Practice Questions
QUESTION 01
An analyst needs to detect when a process spawns cmd.exe, which then runs net.exe, followed by an outbound network connection — all within 2 minutes on the same host. Which rule type should be used?
Correct: Event Correlation (EQL sequence) — This is the classic use case for EQL. Sequences detect ordered chains of events with a by clause (to scope to same host) and with maxspan (2-minute window). KQL cannot express event ordering. Threshold rules count occurrences, not sequences. ML detects anomalies, not specific chains.
QUESTION 02
Review this EQL snippet. What is the purpose of the by host.id clause?
sequence by host.id with maxspan=5m
  [process where process.name == "wscript.exe"]
  [network where destination.port == 443]
Correct: Ensures both events come from the same host. The by clause in an EQL sequence groups events by the specified field value, meaning event 1 and event 2 must share the same host.id to count as a sequence match. Without it, wscript on Host A followed by a network event on Host B would incorrectly trigger the rule.
QUESTION 03
A detection rule has an interval of 5 minutes. Its look-back window is set to now-4m. What problem will this cause?
Correct: Events in the gap may be missed. If the rule runs every 5 minutes but only looks back 4 minutes, there's a 1-minute blind spot. Events that occurred between 4 and 5 minutes ago fall outside the look-back window and are never evaluated. Best practice is to set the look-back slightly longer than the interval (e.g., interval=5m, from=now-6m).
QUESTION 04
A security team wants to suppress duplicate alerts when the same host triggers a brute-force detection rule hundreds of times within an hour. They want only ONE alert per host per hour. Which feature should they configure?
Correct: Alert Suppression grouped by host.name. Alert Suppression deduplicates alerts within a time window by grouping on a field. Suppressing by host.name for 1 hour means only 1 alert fires per host per hour, even if the rule triggers hundreds of times. Exceptions would stop alerting entirely. Increasing the interval would create detection gaps.
QUESTION 05
Which exception type should be used when you want to prevent Elastic Defend from even generating the security event on the endpoint (not just suppressing the alert)?
Correct: Endpoint Exception. Endpoint exceptions are pushed to the Elastic Agent running Elastic Defend. They instruct the agent to not generate the telemetry event at the source. This is the most aggressive form of suppression — the event never reaches Elasticsearch. Rule exceptions and shared exception lists operate at the SIEM layer (post-ingestion) and only suppress alerts, not the underlying events.
QUESTION 06
A rule needs to alert when a user account that has NEVER previously logged in from a specific country is seen doing so. Which rule type is BEST suited for this?
Correct: New Terms rule. New Terms rules are purpose-built for detecting "first-ever seen" values of a field. Configuring it to track source.geo.country_name grouped by user.name means it will alert the first time a user is seen logging in from any new country, using a historical baseline. A Custom Query can't know what's "new" — it would need a static list of countries to exclude.
QUESTION 07
Where in the rule creation wizard do you map a detection rule to its MITRE ATT&CK tactic and technique?
Correct: About rule step. The rule creation wizard has four main steps: Define rule (query/type), About rule (name, severity, risk score, MITRE mapping, tags), Schedule rule (interval, look-back), and Rule actions (notifications). MITRE ATT&CK mapping is configured in the About step under the Threat section.
