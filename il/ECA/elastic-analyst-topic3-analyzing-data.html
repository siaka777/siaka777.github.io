03 Define a Single Metric Machine Learning Job
MLHigh Priority

A Single Metric job is the simplest ML job type in Kibana. It monitors one metric (like request count or response time) over time and learns its normal behavior — then flags deviations as anomalies.

🎯
Exam objective: "Define a single metric, multi-metric, or population Machine Learning job." Know all three types. Single Metric is the most commonly tested — but the exam may ask for any of them.

04 Define a Multi-Metric ML Job
ML
✓ DONE ▾

A Multi-Metric job analyzes multiple detectors at once — for example, monitoring both request count AND response time on the same job. It can also split results by a keyword field (e.g., per host.name).

# Multi-Metric Job — Key Configuration Navigation: ML → Anomaly Detection → Create job → Multi-metric Detectors (add multiple): Detector 1: count # monitors request volume Detector 2: mean(http.response.bytes) # monitors response size Detector 3: high_mean(response_time) # only flags high values Split Field (optional): host.name.keyword # runs all detectors per host separately Bucket Span: 15m # time window for each analysis bucket Datafeed: Index: filebeat-* Query: {"match_all": {}} # or add a filter Result: One job monitors ALL detectors simultaneously → Fewer jobs, correlated anomaly detection
💡
The Split Field (also called "by field") partitions the analysis per entity. If split by host.name, the job independently models each host — so host A's anomaly doesn't mask host B's normal behavior.
⚠️
The bucket span affects how much data is analyzed per bucket. Too short = noisy. Too long = slow detection. Kibana's wizard suggests an appropriate bucket span based on your data volume — accept it if unsure.
05 Define a Population ML Job
ML
✓ DONE ▾

A Population job compares entities to each other — finding the ones that behave differently from their peers. Perfect for security use cases: "which user is doing something unusual compared to all other users?"

# Population Job — Key Configuration Navigation: ML → Anomaly Detection → Create job → Population Detector: Function: count (or mean, sum, etc.) Over field: user.name.keyword ← THE KEY FIELD # "over field" = the entity being compared to the population # Each unique user gets its own model, compared to all users Optional — Partition By: source.geo.country_name # further segment the population Bucket Span: 1h Example Use Cases: — User making unusually many requests vs. other users (count over user.name) — IP downloading unusually large files vs. other IPs (sum(bytes) over source.ip) — Host with unusual error rate vs. other hosts (count over host.name)
Population vs. Single Metric — When to Use Which
ScenarioUseWhy
"Unusual spike in total request count"Single MetricOne aggregate metric, no per-entity comparison needed
"User making far more requests than peers"Population (over: user)Comparing entity to population baseline
"Unusual count AND bytes per host"Multi-Metric + split by hostMultiple correlated metrics per entity
"Rare event — new user agent never seen before"Single Metric (rare detector)Rarity detection, no peer comparison needed

06 Define and Use a Scripted Field
AdvancedKibana
✓ DONE ▾

A Scripted Field is a computed field defined in a Kibana Data View using Painless (Elasticsearch's scripting language). It doesn't exist in Elasticsearch — it's calculated at query time in Kibana. Use it to derive new values from existing fields for display or visualization.

🎯
Exam objective: "Define and use a scripted field for an index." You'll be asked to create one, then use it in Discover or a visualization. Know Painless syntax basics.
Creating a Scripted Field
1
Go to Stack Management → Data Views
Select the data view you want to add the field to.
2
Click the "Scripted fields" tab → Add scripted field
3
Set Name, Language (Painless), Type, and Script
The Type determines how the value is displayed (number, string, date, boolean, etc.)
4
Use the Preview to test before saving
The preview shows sample results from your index — verify the output looks correct.
5
Save — the field now appears in Discover and Visualize

Painless Quick Reference
# Access a field value doc['field_name'].value # Check if field exists first (safe access) if (doc['field_name'].size() > 0) { return doc['field_name'].value; } else { return null; } # Arithmetic doc['bytes'].value / 1048576.0 # bytes → MB doc['end_time'].value - doc['start_time'].value # duration # String operations doc['host.name'].value + ':' + doc['port'].value # concat # Conditional (ternary) doc['status'].value >= 400 ? 'error' : 'success' # Working with .keyword fields doc['url.path.keyword'].value # exact string from keyword field # Accessing text field (requires fielddata: true on the field) doc['message'].value # may need fielddata enabled
⚠️
Use .keyword fields! For string fields, always use doc['field.keyword'].value not doc['field'].value (which requires fielddata enabled on the text field). The .keyword subfield is always directly accessible.
💡
After creating a scripted field, go to Discover and add it as a column — confirm it shows the expected computed values. Then use it in a visualization like any other field.

07 Define and Use a Kibana Space
SpacesKibana
✓ DONE ▾

A Kibana Space is an isolated environment within a single Kibana instance. Each Space has its own saved objects (dashboards, data views, visualizations, ML jobs) and optionally its own feature access settings. Great for multi-team environments.

🎯
Exam objective: "Define and use a Space in Kibana." You may be asked to create a new Space, switch to it, create objects within it, and verify isolation from the Default space.
